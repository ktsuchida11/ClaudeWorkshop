# 第8回 自己改善エージェント ― 想定 Q&A

> **核フレーズ**：「正解があれば毎日賢くなる」
> **題材**：市場レポート（為替など）の日次配信を AI が下書きし、筆者の実レポート（＝正解）と毎日突き合わせて書き方ルールを自動更新する自己改善エージェント。
> **3層構成**：L1（未経験者・全体像）/ L2（日常利用者・運用と実装の勘所）/ L3（深掘り・設計判断と限界）。本編で深追いしない補足はここに退避する。

---

## L1 ― 未経験者向け（全体像・用語）

### Q1-1　そもそも何をするエージェント？
**A.** 市場レポート（為替など）の「全体総括＋本日の注目ポイント」を AI が毎朝下書きし、その日に筆者が実際に配信したレポート（＝正解）と毎昼突き合わせて採点し、翌朝の書き方を少しずつ直していくエージェントです。要点は1つ ―― **「書く役」と「採点する役」を分けて回している**こと。これがシステム全体を貫く設計思想です。

### Q1-2　「正解（gold_standard）」とは何？
**A.** 採点の基準になる唯一の“正解”＝**筆者が実際に配信したレポート**です。AI のドラフトをこの正解と比べることで「どこが足りなかったか」を毎日測れます。逆に言えば、**既に正解がある業務**だからこのループが成立します（「正解を定義できるか？」が出発点）。

### Q1-3　スライドの「書き方ルール（R-NNN）」とは？
**A.** AI が学習で育てる**文体・表現のルール**です。1件＝1ルールで、ファイル上は R-001, R-002… の通し番号で管理し、追加日・更新日・元になった FB・**hit_count（効いた回数）**などのメタ情報が付きます。ペルソナや禁止事項などの固定領域（FIXED）とは分けられ、改善層が追加・更新・削除してよい編集可能領域（EDITABLE）に置かれます。

### Q1-4　人間の手間はどれくらい？
**A.** 日次は基本ノータッチ（自動）です。人間の仕事は、**教師データの手動転送（正解が届かない日）・筆者フィードバックの取り込み・評価基準や憲法の変更承認・メールの最終送信**の4つだけ。AI が「下書きと自己採点」を高速で回し、人間は「方向づけと最終責任」を持ちます。

## L2 ― 日常利用者向け（運用・実装の勘所）

### Q2-1　人間とエージェントの接点は？専用ツールが要る？
**A.** **接点は Gmail だけ**です。専用 UI もダッシュボードも作っていません。AI が Gmail に下書きを作り→人間が確認して送信→筆者がドラフトに朱書き／返信でフィードバック→それを Gmail から取り込んでルール化、という流れ。**「ドラフトの返信に赤を入れるだけ」**で改善指示が学習に入る、自然な業務フローに乗せた設計です。

### Q2-2　なぜメールを自動送信しないの？
**A.** 金融・市場レポートでは誤情報・誤送信のダメージが大きいためです。自動化しても **Gmail の下書き作成まで**で、送信ボタンは人間が押します。送信 API の呼び出しは設計上で明確に禁止しています（送信は人間の最終責任）。

### Q2-3　スコアが低い日は自動で書き直す？
**A.** **書き直しません。** 採点スコアは記録・学習用です。点が低くてもその場で再生成する処理はありません（再生成ループを持つのはファクトチェックの最大2巡だけ）。低スコアの理由はフィードバックに残り、**翌朝の冒頭ステップで「申し送り」として次サイクルの改善に回ります**（採点と改善の責務分離）。

### Q2-4　評価できない日（正解が届かない日）はどうする？
**A.** `no_gold`（教師データ未着）として**採点を保留**し、エラーにはしません。朝のドラフト生成は通常どおり走り、昼の評価だけスキップ。人間が後から正解を手動転送すれば、遡って採点できます。これは「評価情報が定期的に届く」という前提が崩れた日への割り切りです。

### Q2-5　情報収集はどう設計した？汎用の検索？
**A.** 汎用ではありません。**筆者のレポートが何を根拠に書かれているかを精査し、逆算で必要な手段を設計**して、足りない情報をツール／クエリに実装しました。市場価格（M）・ニュース（N）・指標（I）・文脈（B）の4カテゴリで集め、価格は API 化して粒度を確保するなど、「レポートを読む→足りない検索を見つける→道具に落とす」を反復しています。

## L3 ― 深掘り（設計判断・限界）

### Q3-1　評価基準はどうやって作った？恣意的では？
**A.** 恣意的ではなく**過去レポートから逆算**しました。①目的（読み手・成功4条件）から評価軸 S1〜S3 を導出 → ②過去レポート約15本を分析して筆者の分析パターンを抽出 → ③代表1本を「型」として解剖し文章設計の12型を言語化 → ④逆引き検証で「スコアは高いのに筆者は低評価」という盲点を発見し「筆者型 hit 検査」を追加。**基準そのものは人間承認制で固定（FIXED）**し、学習で日々動くのはルール側（R-NNN）だけ。物差しが揺れると学習が壊れるためです。

### Q3-2　評価する AI と書く AI が同じだと“自分に甘く”ならない？
**A.** それを避けるための**層分離**です。Evaluator は**別セッション・別サブエージェント**で、正解（筆者の実レポート）と比べて絶対評価のアンカーで採点します。さらに **Generator は正解（gold_standard）を読めない**制約（カンニング防止）。採点者に媚びる“メタ最適化”を、運用規律ではなく機能境界で防いでいます。

### Q3-3　ルールの過学習（今日だけ効くルールの蓄積）はどう防ぐ？
**A.** **書き方ルールは20件で固定**し、新規追加は必ず**削除・統合とセット**にする「入れ替え制」です。歯止めは3つ ―― ①追加前の **Overfitting 自問**（「今日の正解が無くても他日の品質を上げるか？」）、②**hit_count** で効かないルールを削除候補に、③**Simplicity Criterion**（同品質ならルールは少ない方が勝ち）。入れ替えのトリガーは筆者の直接 FB が最優先、次に週次レビュー（同じ指摘3回以上＝構造的課題）。

### Q3-4　LLM の幻覚（事実誤り）はどう抑える？
**A.** **2層ファクトチェック**です。Layer 1（コード）で、ドラフトの主張が収集データに根拠を持つかを照合し、根拠なしを FAIL 検出。Layer 3（WebSearch）で価格の方向などを外部の独立ソースと照合。FAIL は最大2巡で修正し、残れば末尾に警告を付けて人間判断へ。加えて Generator は収集データ外の事実を書かない制約があります。

### Q3-5　Cron が7日で失効するのに、なぜ止まらない？
**A.** 評価スキルの**最終ステップで毎日、生成・評価の両 Cron を再登録**しているためです（削除→再作成、durable）。実行日が週5回あるので、そのうち1回成功すれば常に7日のバッファを維持できる冗長設計。「失効する仕組み」を逆手に、冪等な再登録で常駐させています。

### Q3-6　このシステムの正直な限界は？
**A.** 主に5点 ―― ①**評価の人間依存**（正解が自動で拾えず手動転送に支えられる＝`no_gold` が常態）②**循環参照の死角**（筆者が AI ドラフトを採用した日は独立した正解が無く、一番の成功日が測れない）③**採点者も LLM** で単一評価のブレが残る ④スコアは「**筆者一致度**」であって絶対品質ではない（下降が必ずしも劣化ではない）⑤20件上限の維持は週次・人手判断のコスト。これらが次回「開発への応用」で評価を自動化したくなる動機にもなります。

### Q3-7　他の業務に転用できる？
**A.** できます。「**生成と評価の分離**」「**正解との差分を翌サイクルに申し送る**」「**ルールを固定枠の入れ替え制で育てて間引く**」「**内部整合＋外部照合の二層チェック**」は、コード生成・サポート文面・要約などドメインに依存しないパターンです。次回はこれを開発工程に移し替えます（→ 第9回）。
