# 番外編「AIエージェントのセキュリティ比較」 FAQ予想問答集

質疑応答に備えた、想定問答集です。番外編のテーマは「**GitHub Copilot 拡張 vs Claude Code 拡張 vs Claude Code DevContainer**を、**Windows／エンタープライズ環境を前提に**セキュリティ／ガバナンスで比較する」。

過去3回と同じ3層構造で整理：

- **L1 未経験者から出そうな質問**：Copilotとの違い、WSL、サンドボックスなど基本概念
- **L2 日常利用者から出そうな質問**：設定・運用・WSL2構築・組織配布
- **L3 深掘り質問**：改竄耐性・ネットワーク境界・限界・他エージェント比較

各質問には **短い回答（30秒以内）** と、余裕があれば添える **補足** をセット。深入りしそうな質問は「本編の第N回で扱います」と引き取って時間を守る型を推奨。

> **このFAQの大前提（最重要）**
> Copilotにも Claude Code にも、サンドボックスも Hooks も**両方ある**。「片方だけの機能」という誤解で答えないこと。違いは「**止め方**（人の承認 vs ルールで決定論的）」と「**周辺の固め方**（組織配布・改竄耐性・隔離レベルの選べる幅）」にある。

## L1：未経験者から出そうな質問

### Q1-1. Copilot と Claude Code、セキュリティ的にはどっちが安全なんですか？

**短答**：**「どっちが安全」ではなく「止め方が違う」**です。Copilotは「人が承認ゲートで止める」前提（柔軟だが best-effort）、Claude Codeは「許可ルールでツール層が機械的に止める」前提（決定論的）。組織で一貫させたいなら後者の考え方が向きますが、どちらもサンドボックスで隔離すれば最終的な硬さは出せます。今日の結論は「ツールの優劣」ではなく「**どの層で・どう止めるかを設計する**」です。

**補足（求められたら）**：Copilotは承認に倒すか自動承認に倒すかを選べる。自動承認(Bypass)に倒すほど守りは薄くなる。

---

### Q1-2. Copilotにもサンドボックスってあるんですか？Claude Code だけかと思ってました

**短答**：**両方あります**。Copilotは Default Approvals モードで使うとサンドボックスが既定で適用される標準挙動になっています（v1.118 以降）。Claude Code も `/sandbox` で有効化できる。どちらも内部は同じ系統（Linux／WindowsではWSL2 で bubblewrap）の OS レベル隔離です。ただし**Windowsの場合、ネイティブ（PowerShell直）では効かず WSL2 が前提**（詳細は Q1-3・Q1-4）。今日のスライドで「Copilot＝サンドボックス無し」と読めたら、それは誤解なので訂正します。

**補足**：ただしCopilotのサンドボックスは「ターミナルコマンドだけ」が対象で、ファイル編集ツール（read/edit/write）は別管轄（権限システム側）。

---

### Q1-3. WSL2 って何ですか？なんでセキュリティの話に出てくるんですか？

**短答**：WSL2は**Windows上で動く本物のLinux環境**です。なぜ出てくるかというと、**Windowsにはネイティブのサンドボックスが無い**から。サンドボックスの実体（bubblewrap）はLinuxカーネルの機能を使うので、Windowsで隔離を効かせるには WSL2 のLinux側でエージェントを動かす必要があります。CopilotもClaude Codeも、Windowsでは WSL2 が事実上の正攻法です。

**補足**：WSL1は非対応（必要なカーネル機能が無い）。`wsl -l -v` で VERSION 2 か確認。

---

### Q1-4. Windowsでそのまま（WSL使わず）Copilotを使ったら、サンドボックスは効かないんですか？

**短答**：**効きません**。ネイティブWindows（PowerShell/cmd直）では bubblewrap が動かないので、サンドボックスの基盤そのものが無い。設定で `enabled` を立てても「箱が無い」状態です。しかも危険なのは、サンドボックス前提で自動承認に倒していると、**隔離が無いのに確認だけ省かれてホスト直で危険コマンドが走り得る**こと。Windowsで隔離が要るなら WSL2 か DevContainer に移すのが前提です。

**補足**：付録（Appendix）に「WSL2なしでON にしたら何が起きるか」を1枚入れてあります。実際にWSL併用の中途半端構成でホームディレクトリが消えた報告（VS Code issue #316120）もある。

---

### Q1-5. 「Hooks」って何ですか？これも Claude Code だけの機能ですか？

**短答**：**これも両方あります**。Hooksは「AIの判断とは別に、節目で必ず走る決定論的なスクリプト」で、例えばツール実行の直前（PreToolUse）に危険コマンドをブロックできます。実はCopilotもサポートしていて、しかも**設定フォーマットはClaude Codeと互換**（PreToolUse / PostToolUse など同じ名前）。CLAUDE.md などの「指示」が助言なら、Hooksは「強制」です。

**補足**：Claude Code版の Hooks は本編第3回で詳しく扱っています。番外編では「Copilotにも同等のものがある」ことが主眼。

## L2：日常利用者から出そうな質問

### Q2-1. WindowsでVS Code + Copilot + WSL2 + Sandbox の構成、最短で組むには？

**短答**：5ステップです。①PowerShell(管理者)で `wsl --install` → 再起動、②拡張「WSL」を入れて WSL: Connect（左下が「WSL: <distro>」表示）、③CopilotをWSL側にインストールしエージェントモード有効化、④settings.json で `"chat.agent.sandbox.enabled": "on"`（or `"allowNetwork"`）、⑤必要なOS依存(bubblewrap等)が無ければVS Codeが導入を提案するので承認。所要15分ほど。Appendixに設定例を載せています。

**補足**：プロジェクトは WSL側（例 `~/work`）に置く。`/mnt/c` のWindows側はI/Oが遅い。

---

### Q2-2. Copilotのサンドボックスの設定、`chat.tools.terminal.sandbox` じゃないんですか？

**短答**：**それは古い名前です**。当初は `chat.tools.terminal.sandbox.enabled` でしたが、v1.114〜1.115頃に `chat.agent.sandbox.*` に統一されました。今は `chat.agent.sandbox.enabled` で、値は文字列の `off`（既定・無効）／`on`（ファイル＋通信の完全分離）／`allowNetwork`（ファイルのみ分離で通信は自由）の3択です。`true`/`false` やモード別キーではなく**この文字列値**が正しい（公式Docs準拠）。古いバージョンを使っている人は旧名のままなので注意。

**補足**：スライドのフッターは「2026年5月時点」。設定名は変わりやすいので、公式Docs（code.visualstudio.com）で最新を確認するのが安全。

---

### Q2-3. URLの承認ゲートがあるなら、エージェントが勝手にWebに繋ぐ心配はないですよね？

**短答**：**そこに落とし穴があります**。URLゲートが守るのは `#fetch` ツールやブラウザだけ。エージェントがターミナルで `curl` や PowerShell の `iwr` を打つと、それは別管轄（ターミナル承認＝best-effort）で抜け得る。**通信を決定的に止めるのはサンドボックスのネットワーク隔離**で、許可ドメイン以外を遮断します。「URLゲートがあるから安心」ではなく「サンドボックスまで重ねて初めて締まる」が正解。

**補足**：Claude Code側も同じ構図。WebFetchを許可制にしても Bash の curl は別なので、`Bash(curl:*)` を deny ＋ サンドボックスのドメイン許可制で塞ぐ。

---

### Q2-4. ネットワーク制限を「ドメイン」で書くなら、IPアドレス直打ちは防げるんですか？

**短答**：**そこが両者共通の弱点です**。ガードがドメイン許可制を前提にしているので、IPアドレス直指定・DNSリバインディング・標準的でないDNS解決をするプロセス（Node の fetch など）は抜け得る。これはツールの優劣ではなく「ドメイン名で止める方式」の構造的な穴です。本当にIPレベルで固めたいなら、**DevContainer の iptables（default-deny ＋ 送信先IP/ポートで許可）**まで降ろすのが確実。

**補足**：過去にSOCKS5プロキシで `evil.com\0.allowed.com` のようにヌルバイトを挟んで許可リストを通り抜けるバイパスがあったが、Claude Code（srt）ではパッチ済み。

---

### Q2-5. ユーザーが自分で settings.json を書き換えたら、組織のポリシーは無効化されちゃうんですか？

**短答**：**方式が違います**。Copilotは「ユーザー設定ファイルは編集できるが、ポリシー項目は書いても効かない（無視される）」方式。Claude Codeは「ポリシーを置いたファイル自体を、管理者権限が無いと書き換えられない」方式（Windowsなら HKLM や `C:\Program Files\ClaudeCode`）。さらにClaude Codeのサーバー管理設定なら、端末にそもそも編集対象のファイルが無い。

**補足**：共通の急所は、ユーザーがローカル管理者権限を持っているとどちらも置き場所を書き換えられること。だから**日常作業は非管理者ユーザー**が前提。

---

### Q2-6. Windowsで組織のポリシーを配布する「GPO」って何ですか？

**短答**：GPO＝Group Policy（グループポリシー）で、**Active Directory配下のWindows端末に設定を一括配布・強制する仕組み**です。管理者が定義すると各PCのレジストリ（HKLM配下）に自動適用され、ユーザーは変更できません。Copilotのポリシーは ADMX テンプレート経由で `HKLM\SOFTWARE\Policies\Microsoft\VSCode` に配布します。Intune でも配布できますが、当該レジストリは予約されていて取り込めないので Proactive Remediation（SYSTEM権限のPowerShell）で回避する、という小技があります。

**補足**：HKLM＝HKEY_LOCAL_MACHINE。レジストリの「PC全体（全ユーザー共通）」の領域で、書込に管理者権限が要る。個人用は HKCU。

## L3：深掘り質問（想定しておくと詰まらない）

### Q3-1. 「ターミナル実行」「Bashツール」「子プロセス」、結局どれが一番強く守られるんですか？

**短答**：**サンドボックスが効いていれば、どれも同じく決定的に守られます**。なぜなら隔離はOSカーネルレベルなので、サンドボックス内で起動した子プロセスも境界の内側に閉じ込められるから。プロンプトインジェクションで承認を突破されても、ファイルと通信は遮断される。逆に承認ゲート（best-effort）は突破され得る。だから「承認に頼る」より「サンドボックスで囲む」のが構造的に強い、というのが今日の核です。

**補足**：CopilotもClaude Codeも「子プロセスまで継承」を明示している。違いは成熟度（Claude Code＝GA・srtがOSS、Copilot＝WSL2基盤で標準化中）。

---

### Q3-2. Copilotのサンドボックスは「ファイル編集ツールには効かない」と言いましたが、それは穴では？

**短答**：**鋭い指摘で、運用上の注意点です**。Copilotのサンドボックスはシェルのサブプロセス（ターミナルコマンド）にのみ適用され、組み込みの read/edit/write ファイルツールはサンドボックスを通らず、VS Codeの権限システムを直接使います。つまり「ターミナル経由の破壊」はサンドボックスで止まるが、「ファイル編集ツール経由」は権限システム側で止める二本立て。両方を設定して初めて穴が無くなります。

**補足**：Claude Code側は Read/Edit の deny ルールがサンドボックス境界に合成されるので、ここはやや一貫している。

---

### Q3-3. 「サーバー管理設定が最強」と言うけど、それはサインインしないと効かないのでは？

**短答**：その通りで、**そこが弱点でもあります**。Claude Codeのサーバー管理設定（管理コンソールからリモート配信）は最も改竄耐性が高い（端末にファイルが無い）が、**組織サインイン後にしか適用されない**。サインイン前の段階を固めるには、結局 MDM やOSファイル（HKLM / Program Files）での配布を併用する必要があります。「サーバー管理＋エンドポイント（MDM）＋fail-closed（取得失敗で起動中止）」の三段構えが実務解です。

**補足**：Copilotも同様で、GitHub組織側（サーバー）とVS Code側（レジストリ）の二層を両方そろえて初めて穴が無い。

---

### Q3-4. Copilot と Claude Code、エンタープライズ配布の「制御プレーン」はどう違う？

**短答**：**集約度が違います**。Copilotは「VS Codeデバイスポリシー（Intune/GPO/MDM）」＋「GitHub組織/Enterpriseポリシー」の**二層を両方**そろえる必要がある。Claude Codeは「サーバー管理（管理コンソール）」＋「エンドポイント（MDM/OSファイル）」を**一系統**に集約でき、SCIM（ID連携）でグループ単位に配布、WSL側にも `wslInheritsWindowsSettings` で継承できる。「穴を作らないために何系統そろえるか」が違いです。

**補足**：SCIM＝システム間でユーザー/グループ情報を連携する標準。これでグループ単位にポリシーを配れる。

---

### Q3-5. 他のエージェント（OpenAI Codex など）と比べてWindows対応はどうなんですか？

**短答**：**Codexだけはネイティブ Windows でもサンドボックスを持っています**。Codexは Windows で PowerShell 実行時にネイティブWindowsサンドボックスを使い、WSL2 実行時はLinuxサンドボックス実装を使い分ける。一方 Copilot と Claude Code は現状ネイティブWindows非対応で、**WSL2 が必要**。「Windowsネイティブで隔離したい」が要件なら、この差は効いてきます。ただし実務では WSL2 か DevContainer に寄せるのが結局一番固いので、大勢に影響はしません。

**補足**：他社比較は今日の範囲を超えるので、「考え方として受け取ってほしい」と引き取るのが安全。

---

### Q3-6. 結局、組織で標準化するならどの構成を推すんですか？

**短答**：**「決定的な隔離 ＋ ポリシーのソース保護 ＋ 監査」の三点セット**を推します。具体的には、①高リスク作業は WSL2 サンドボックス or DevContainer（コンテナ）で隔離、②ポリシーは MDM/OSファイル＋（Claude Codeなら）サーバー管理で配布し fail-closed、③監査ログ＋ConfigChange検知で変更を捕捉。日常作業は非管理者ユーザー＋最小権限。ツール単体ではなく**この型を組織に敷く**のが答えです。

**補足**：監査（「残す」）は本編第6回でLangfuseと合わせて扱う予定。番外編は「分ける・防ぐ」を組織レベルに引き上げる回。

## 質疑で時間を使いすぎた時の撤退ライン

- 質疑は**5分厳守**（過去3回と同じ）
- L3系の深掘り（改竄耐性・制御プレーン比較）は1問で5分使い切るリスクあり → **簡潔な答え＋「深掘りは個別で」**の型

- 質問が出ない時の呼び水：「よく聞かれるのは『Copilotにもサンドボックスあるの?』ですが…」（Q1-2）で自発的に話題提示

## 過去回FAQ との差分ポイント

第1回は「脅威・リスクの棚卸し」、第2回は「環境の分離（分ける）」、第3回は「settings.json で防ぐ」が中心でした。**番外編は「Copilotとの横並び比較」と「Windows／エンタープライズ前提」**が中心なので、FAQも以下の論点が新しい：

- **Copilotにも同等機能がある**ことの繰り返しの明確化（Q1-2, Q1-5, Q3-2）＝優劣論争を避ける軸
- **WindowsでのWSL2前提**という固有事情（Q1-3, Q1-4, Q2-1）
- **設定キー名の改名**という実務的な落とし穴（Q2-2）
- **改竄耐性・制御プレーンの比較**（Q2-5, Q3-3, Q3-4）＝第3回の「防ぐ」を組織レベルに拡張
- **IP直アクセス・ファイルツール別管轄**という両者共通の限界（Q2-4, Q3-2）
- **他エージェント比較**（Q3-5）

特に「どっちが安全？」という素朴な問いに**優劣ではなく構造の違いで答える**のが、この番外編FAQの一番のポイント。
