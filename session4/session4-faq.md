# 第4回「自動化・共有」 FAQ予想問答集

質疑応答5分間に備えた想定問答集。第4回のテーマは「**【防ぐ】の延長 ― Skills / MCP / スラッシュコマンド / コンテキストエンジニアリング**」。

3層構造で整理：

- **L1 未経験者から出そうな質問**：Skills・MCP・コマンドの基本概念
- **L2 日常利用者から出そうな質問**：設定の細部・運用・トークン節約
- **L3 深掘り質問**：コンテキストエンジニアリング・組織展開・アーキテクチャ

---

## L1：未経験者から出そうな質問

### Q1-1. Skills と CLAUDE.md、結局どう使い分けるんですか？

**短答**：**常に必要か、たまに必要か**で分けます。CLAUDE.mdは「このプロジェクトでは常にESLintを使う」「コミットメッセージはConventional Commitsで」のような**プロジェクト全体のルール**。Skillsは「Dockerfileを書く時はこの手順で」「PRレビューの時はこのチェックリストで」のような**特定のタスクの知識・手順**。

CLAUDE.mdは毎セッションで全文読み込まれるのでトークンを常に消費する。Skillsは必要な時だけ読み込まれるのでゼロから始まる。CLAUDE.mdが200行を超えそうなら、一部をSkillsに切り出すのが目安。

---

### Q1-2. MCP って何の略ですか？何ができるんですか？

**短答**：**Model Context Protocol**の略。Claude Codeと外部ツール（GitHub、Jira、Slack、DB等）を接続するための標準プロトコル。MCPサーバーを追加すると、Claude Codeが直接GitHubにPRを作ったり、Jiraにチケットを起票したりできるようになる。

ただし**2つの注意点**：
1. MCPツール定義がコンテキストウィンドウを圧迫する（20kトークン超えも）
2. `.mcp.json`がリポジトリにコミットされるため、悪意あるリポジトリ経由で攻撃される可能性がある

だから`enableAllProjectMcpServers: false`が鉄則。

---

### Q1-3. スラッシュコマンドって、ただのショートカットですか？

**短答**：**ショートカット以上のもの**です。スラッシュコマンドはMarkdownファイルに書かれた「プロンプトテンプレート」で、`$ARGUMENTS`で引数を受け取れる。例えば`/review`コマンドに「このファイルをレビューして、セキュリティ問題を重点的にチェック」というプロンプトを書いておけば、毎回同じ品質のレビューが得られる。

Skillsとの違い：Skillsは**Claude Codeが自動で判断して読み込む**（model-invoked）。コマンドは**ユーザーが明示的に呼ぶ**（user-invoked）。「自動で発火してほしい」ならSkills、「自分のタイミングで使いたい」ならコマンド。

---

### Q1-4. Plugin って、チームの他のメンバーにどうやって渡すんですか？

**短答**：**2つの方法**があります。
1. **GitHubリポジトリ経由**：Plugin用のリポジトリを作り、`claude plugin marketplace add <owner>/<repo>` でチームメンバーがインストール
2. **プロジェクトリポジトリ内に配置**：`.claude/plugins/` にPluginフォルダを置いてコミット → クローンした人に自動適用

第3回のsettings.json（セキュリティ設定）もPlugin化してチーム全体に展開できる。セキュリティ + Skills + コマンドをまとめて「うちのチームの標準設定」として配布する使い方が実用的。

---

### Q1-5. コンテキストエンジニアリングって、難しそうな名前ですが何ですか？

**短答**：**Claude Codeの「頭の中」を整理する技術**です。Claude Codeには「コンテキストウィンドウ」という作業メモリがあって、容量に限りがある。何も管理しないと古い情報で埋まって精度が落ちる。

3つの戦略で対処する：
1. **圧縮**（`/compact`）：要約して容量を空ける
2. **構造化ノート**：重要情報をファイルに書き出して、コンテキストから追い出す
3. **サブエージェント**：独立した作業メモリで別タスクを処理させる

一番大事なのは戦略2で、「コンテキストウィンドウの中身は揮発する。ファイルは残る」という原則。

## L2：日常利用者から出そうな質問

### Q2-1. Skill の description をうまく書くコツはありますか？発火しないことがある

**短答**：**「少し強めに書く」のがコツ**です。Anthropicの公式ドキュメントでも「Claude has a tendency to undertrigger skills」と認めている。descriptionは控えめに書くと発火しない。

悪い例：`"description": "How to create a dashboard"`
良い例：`"description": "How to create a dashboard. Use this skill whenever the user mentions dashboards, data visualization, metrics display, charts, or wants to see any kind of data summary, even if they don't explicitly say 'dashboard'."`

ポイント：
- **トリガーすべきキーワードを列挙**する
- **「明示的に言わなくても使え」**と書く
- `globs`（ファイルパターン）も併用すると精度が上がる（例：`"globs": ["*.dockerfile", "Dockerfile*"]`）

---

### Q2-2. MCP→Skills 変換の「80-98%削減」って、具体的にどういう計算ですか？

**短答**：MCPサーバーを接続すると、そのサーバーが提供する**全ツールの定義（名前・パラメータ・説明文）**がコンテキストウィンドウに常駐する。例えばGitHub MCPサーバーなら30以上のツール定義で10-20kトークンを消費する。

Skillsに変換すると、実際に使うパターン（例：「PRを作成する手順」）だけをSKILL.mdに書くので、数百トークンで済む。しかもSkillsは必要時のみ読み込まれるので、使わない時はゼロ。

計算：MCP常駐 20kトークン → Skill必要時のみ 0.4-4kトークン = **80-98%削減**

ただし**リアルタイムデータ参照が必要なMCP（DB検索、ライブステータス）はSkillsに変換できない**。定型操作パターンだけが対象。

---

### Q2-3. enableAllProjectMcpServers と enabledMcpjsonServers の違いがよく分からない

**短答**：

- **`enableAllProjectMcpServers: true`**：プロジェクトの`.mcp.json`に書かれた**全MCPサーバーを問答無用で有効化**する。悪意あるリポジトリをクローンしたら即座にそのサーバーが動く。**絶対にtrueにしない**。
- **`enabledMcpjsonServers: ["github", "jira"]`**：`.mcp.json`の中から**使いたいサーバーだけ名前でホワイトリスト**する。ここに書いていないサーバーは無視される。

設定例：
```json
{
  "enableAllProjectMcpServers": false,
  "enabledMcpjsonServers": ["github", "jira"]
}
```

`.mcp.json`に`malicious-server`が仕込まれていても、`enabledMcpjsonServers`に入っていないから無視される。

---

### Q2-4. /compact と /clear、どっちをいつ使えばいいですか？

**短答**：

| | `/compact` | `/clear` |
|---|---|---|
| やること | コンテキストを要約して圧縮 | コンテキストを全消去 |
| CLAUDE.md | 再読み込みされる | 再読み込みされる |
| ファイル | 影響なし | 影響なし |
| 使うタイミング | 同じタスクの途中で容量が詰まった時 | 完全に別のタスクに切り替える時 |
| 注意点 | 必要な情報も要約で消える可能性 | 全て失われる |

**鉄則**：どちらを使う前にも、**まずアウトプットをファイルに残す**。`docs/progress.md`に作業ログを書き出してから`/compact`する。そうすれば要約で消えても、ファイルを読み直せば復帰できる。

---

### Q2-5. 構造化ノート（docs/plan.md 等）は CLAUDE.md に書くのとどう違う？

**短答**：**更新頻度と性質が違います**。

- **CLAUDE.md**：ほぼ変わらないルール・規約（コーディング規約、ビルドコマンド等）
- **docs/plan.md**：タスクごとに変わる作業計画
- **docs/decisions.md**：都度追記される設計判断
- **docs/progress.md**：毎日更新される作業ログ

CLAUDE.mdに「docs/plan.mdを参照して作業を進めて」と書いておけば、新セッションでもClaude Codeが自動的にplan.mdを読んで文脈を引き継ぐ。

ポイントは**「CLAUDE.mdは指さし棒、docs/は実際の地図」**という関係。

---

### Q2-6. サブエージェントに渡すタスクの粒度はどのくらいが適切？

**短答**：**「結果がファイル1つで返せる」粒度**が目安。サブエージェントは独立したコンテキストで動くので、結果を親に返す手段がファイルしかない。だからタスクが「テスト結果のレポート」「影響範囲の一覧」「コードレビューのコメント」のように、1ファイルにまとまる単位で切るのが良い。

逆にNGなのは「このプロジェクト全体をリファクタリングして」のような粒度。サブエージェントのコンテキストウィンドウも有限なので、大きすぎるタスクは失敗する。

---

### Q2-7. .mcp.json に書いたAPIキーがgitにコミットされたら事故ですよね？どう防ぐ？

**短答**：**3つの防御層**で守ります。

1. **環境変数参照**：`.mcp.json`内で`"apiKey": "${GITHUB_TOKEN}"`のように`${VAR}`記法を使う。キー本体は`.env`やOSの環境変数に持つ
2. **.gitignore**：`.env`をgitignoreに追加。`.mcp.json`自体はコミットしてOK（キーが入っていなければ）
3. **git-secrets / pre-commit hook**：コミット時にシークレットパターン（`ghp_`, `sk-`等）をスキャンしてブロック

万が一コミットしてしまったら、即座にキーをローテーション（再発行+旧キー無効化）。gitの履歴からも`git filter-branch`で削除する。

## L3：深掘り質問

### Q3-1. Agent Skills のオープンスタンダード化って、具体的にどういうこと？

**短答**：Anthropicが2025年12月にAgent Skills仕様をオープンスタンダードとしてリリースしました。`SKILL.md`のfrontmatterフォーマット（name/description/globs等）が標準化されていて、**OpenAIのCodex CLIも同じ形式を採用**している。つまり、Claude Code用に作ったSkillがCodex CLIでもそのまま使える可能性がある。

実務的な意味：Skillsに投資する価値が高い。ベンダーロックインになりにくい。

---

### Q3-2. Skill Creator の Eval モードって何ですか？今日の実演では使わなかったけど

**短答**：**Skillのテストフレームワーク**です。4つのモードのうちの1つ：
- **Create**：Skillを新規作成（今日の実演で使用）
- **Eval**：テストプロンプトに対してSkillが正しく発火・動作するか検証
- **Improve**：テスト結果に基づいてSkillを自動改善
- **Benchmark**：複数バージョンのSkillをA/B比較（統計的信頼度付き）

Evalが特に便利なのは**モデル更新時**。新しいClaudeモデルにアップデートした時にSkillが同じ品質で動くか、Evalで回帰テストできる。ソフトウェアテストと同じ発想。

---

### Q3-3. コンテキストエンジニアリングの構造化ノート、チームで使う場合のルールは？

**短答**：チームで使うなら以下の3つを決めておく。

1. **ファイルの場所と命名規則**：`docs/`配下、ファイル名は`plan-<feature名>.md`のように命名
2. **更新タイミング**：作業開始前にplan作成、設計判断時にdecisions追記、作業終了時にprogress更新
3. **CLAUDE.mdへの参照ルール**：「このプロジェクトでは`docs/`配下のファイルを必ず参照して作業を進めてください」と記載

チーム全員が同じ構造化ノートを使えば、誰かが書いたplan.mdを別の人のセッションでも引き継げる。**「AIとの仕事の型」を標準化する第一歩**（第5回のテーマにつながる）。

---

### Q3-4. MCPサーバーを自作する場合のセキュリティ注意点は？

**短答**：自作MCPサーバーは**実質的にClaude Codeに任意のツールを渡す**ことなので、以下に注意：

1. **入力バリデーション**：Claude Codeからのリクエストパラメータを必ず検証。SQLインジェクション等の一般的な攻撃ベクトルに注意
2. **最小権限**：MCPサーバーが接続するDB/APIの認証情報は最小権限にする（第2回のIAM最小権限と同じ考え方）
3. **ログ記録**：MCPサーバーの全リクエスト/レスポンスをログに残す（第6回「残す」に接続）
4. **レート制限**：無限ループでAPIを叩き続けるのを防ぐ
5. **テスト**：Skill Creator的なEvalの発想で、MCPサーバーの動作テストを書く

---

### Q3-5. コンテキストウィンドウのサイズって、どのくらい余裕があるんですか？

**短答**：Claude Sonnet 4.6 / Opus 4.6 は**200kトークン**のコンテキストウィンドウ。大きいように見えるが、以下で消費される：

- システムプロンプト + CLAUDE.md：数千トークン
- MCPツール定義：接続数次第で10-30kトークン
- 会話履歴：ターンごとに数百〜数千トークン
- ファイル内容：参照ファイル1つで数千トークン

実感としては**10-20ターン程度で圧迫が始まる**。だからMCP→Skills変換（常駐トークン削減）と/compact（不要な履歴の圧縮）が効く。

---

### Q3-6. 今日の内容を全部実装すると、プロジェクトの .claude/ フォルダはどういう構成になる？

**短答**：第3回（settings.json）+ 第4回（Skills/コマンド）を合わせた完全構成：

```
.claude/
├── settings.json        ← 第3回：Permission/Sandbox/Hooks/NW allowlist
├── settings.local.json  ← 個人調整用（.gitignore対象）
├── hooks/
│   └── pretool-guard.sh ← 第3回：PreToolUse Hook
├── skills/
│   ├── todo-lister/     ← 第4回：実演で作成
│   │   └── SKILL.md
│   └── pr-reviewer/     ← 追加Skill
│       └── SKILL.md
├── commands/
│   ├── deploy.md        ← 第4回：デプロイ手順
│   └── review.md        ← 第4回：レビューチェックリスト
└── CLAUDE.md            ← プロジェクトルールの本体
```

この構成をPlugin化すれば、チーム全員に一括配布できる。

### Q3-7. 今日学んだ5つの武器、優先順位をつけるとしたら？

**短答**：**投資対効果の順**で並べると：

1. **CLAUDE.md の整理**（5分で効果大）：まだ書いていないなら最初にやる
2. **settings.json の配置**（10分で効果大）：第3回のベストプラクティス版をコピー
3. **構造化ノート運用の開始**（0分で効果中）：docs/plan.md を1つ作るだけ
4. **Skills の作成**（30分で効果中）：Skill Creatorで1つ作ってみる
5. **MCP→Skills変換**（1時間で効果大）：MCPを使っている人だけが対象

1-3は**今日中にできる**。4は今日の実演で体験済み。5はMCPを使っていなければ不要。
