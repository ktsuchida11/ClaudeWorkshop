# 第6回「ログ/トレース」 FAQ予想問答集

質疑応答5分間に備えた想定問答集。第6回のテーマは「**3原則の【03 残す】の実装編 ― 動いた≠OK**」。

3層構造で整理：

- **L3 深掘り質問**：ツール選定・評価設計・組織展開の判断（7問+メタ戦術）

## L1：未経験者から出そうな質問

### Q1-1. 「LLM Observability」って従来のログ監視と何が違うんですか？

**短答**：従来の監視は「**サーバーが落ちていないか、レスポンスが遅くないか**」を見るもの。LLM Observabilityは「**AIがどう考えて、何を出力して、その品質はどうだったか**」を見るもの。

従来のシステムは同じ入力に同じ出力を返すから、エラーレートやレイテンシで十分だった。でもClaude Codeを含む生成AIは同じ指示でも毎回違う結果を返す。「動いたけど品質が悪い」「途中で間違った判断をした」を検知するには、推論過程を丸ごと記録する仕組みが要る。

---

### Q1-2. Langfuseは無料で使えますか？

**短答**：**OSSなのでセルフホストすれば無料**。AWSのVPC内に自分で構築すれば、ソフトウェアのライセンス料はかからない。かかるのはインフラ代（EC2、RDS、S3等）のみ。

クラウド版（Langfuse Cloud）もあり、少量なら無料枠がある。ただし金融機関など厳しいセキュリティ要件がある場合は、データを外部に出せないのでセルフホスト一択になる。

---

### Q1-3. Claude Codeのトレースって、何が記録されるんですか？

**短答**：Claude Codeが1つの指示に対して実行する**全ステップ**が記録される。具体的には：

- どのファイルを読んだか（Read: src/auth/login.ts）
- CLAUDE.mdを参照したか
- どんな計画を立てたか（Planning）
- どのファイルをどう編集したか（Edit）
- どのコマンドを実行したか（Bash: npm run test）
- テストが成功したか失敗したか
- リトライしたか、何回ループしたか
- 各ステップのトークン数・コスト・所要時間

これが1本のツリー構造で可視化される。「なぜうまくいかなかったのか」を事後的にデバッグできるようになる。

---

### Q1-4. 「Trace→Evaluate→Improve」って何ですか？

**短答**：Langfuseの3つのコア機能を表すサイクル。

1. **Trace（可視化）**：Claude Codeの推論過程を記録する
2. **Evaluate（評価）**：記録したトレースに品質スコアを付ける（LLM-as-a-Judgeやユーザーフィードバック）
3. **Improve（改善）**：スコアデータを基にプロンプトを修正し、Prompt CMSでバージョン管理・デプロイする

このサイクルを回すことで「推測」ではなく「データ」に基づいてAIの品質を継続的に改善できる。第5回で学んだ「計画→実装→記録」のループを**システムレベルで自動化**したもの。

---

### Q1-5. LLM-as-a-Judgeって、AIにAIを評価させて大丈夫なんですか？

**短答**：**完璧ではないが、人間だけでやるよりはるかにスケールする**。人間が目視でチェックできるのは月に数件〜数十件だが、LLM-as-a-Judgeなら全件に対して定量スコアを付けられる。

重要なのは「レビューの代わり」ではなく「**品質の変化に気づく仕組み**」として使うこと。スコアが急に下がったら人間が詳しく調べる、というアラート的な使い方が実践的。自動評価 + ユーザーフィードバック + ドメイン専門家の手動アノテーションを**3つ組み合わせる**のがベストプラクティス。

## L2：日常利用者から出そうな質問

### Q2-1. Claude CodeのトレースをLangfuseに送るには、具体的に何をすればいいですか？

**短答**：**Claude CodeのAPI呼び出しをLangfuseのSDKでラップする**だけ。Anthropic Messages APIを直接呼ぶ代わりにLangfuseのラッパーを経由させる。数行のコード変更で導入できる。

```python
from langfuse import Langfuse
langfuse = Langfuse()

trace = langfuse.trace(name="claude-code-session", user_id="koji")
# ここで通常通りClaude APIを呼ぶ（traceにステップが自動記録される）
```

OpenTelemetryネイティブなので、既にOTel対応の監視基盤がある場合はそちら経由でも可能。

---

### Q2-2. トレースを取るとオーバーヘッド（遅延やコスト）はどのくらい増えますか？

**短答**：**Langfuseのオーバーヘッドは約15%**（公式ベンチマーク）。主にトレースデータの送信にかかる。ただしこれはAPI呼び出し自体の時間に比べればごく小さい。Claude CodeのAPI呼び出しが数秒かかるのに対して、トレース送信は数十ミリ秒。

コスト面では、Langfuseのセルフホスト環境（ClickHouse + PostgreSQL + S3）のインフラ費用が追加になるが、月数千円〜数万円レベル。Claude Code自体のAPI費用に比べれば微小。**「計器なしの飛行」のリスクに比べれば安い投資**。

---

### Q2-3. 第3回のHooksと今日のトレース、どういう関係ですか？

**短答**：**Hooksは「防ぐ」、トレースは「残す」。2つが連動する**のがポイント。

- Hooksが危険なコマンドをブロックしたとき、その「ブロックした」という事実もトレースに記録される
- つまりトレースを見れば「Claude Codeが何を実行しようとして、何がブロックされたか」の完全な監査証跡になる
- 第1回で学んだ3原則の「防ぐ」と「残す」がシステムレベルで繋がる

逆に言えば、Hooksなしでトレースだけ取っても「危険なことをやった後に気づく」だけ。**Hooks（予防）+ トレース（記録）の両方揃って初めて安全な運用**。

---

### Q2-4. Prompt CMSとCLAUDE.mdの違いがよく分かりません

**短答**：**管理対象とデプロイ方法が違う**。

| | CLAUDE.md | Prompt CMS |
|---|---|---|
| 対象 | Claude Codeへの行動指針 | LLMアプリのシステムプロンプト |
| 管理 | Gitリポジトリにコミット | LangfuseのUI上でバージョン管理 |
| デプロイ | コード変更が必要 | API経由で即座に反映（コード変更不要） |
| ロールバック | git revert | UI上でワンクリック |
| 利用者 | エンジニア | エンジニア＋ドメイン専門家 |

CLAUDE.mdは第3回で学んだ「Claude Codeの行動指針」。Prompt CMSは「LLMアプリケーションのプロンプトをバージョン管理する仕組み」。レイヤーが違うので**併用するもの**。

---

### Q2-5. トレースデータの量が膨大になりませんか？ストレージは大丈夫？

**短答**：**確かに増える。だからこそLangfuseのアーキテクチャが重要**。

- **ClickHouse**が高速な分析用ストレージとして機能し、大量トレースの検索・集計が高速
- **S3**に長期保存用ログを退避できるので、コストを抑えながら全トレースを保持可能
- PIIマスキングを入れることで、保存するデータ量自体も削減できる
- 実運用では**サンプリング**（全リクエストではなく一定割合だけトレース）も有効

「全部残す」ではなく「**必要な粒度で構造化して残す**」のが第6回で伝えたかったこと。

---

### Q2-6. スライドに出てきた4ツール（Langfuse/LangSmith/Braintrust/Helicone）、もう少し詳しく選び方を教えてください

**短答**：**判断軸は3つ**。

1. **データ統制**: データを自社内に留める必要があるか → あるならLangfuse（セルフホスト可能なOSSは現状これだけ）
2. **フレームワーク依存**: LangChainに特化していいか → よいならLangSmith。汎用性が必要ならLangfuse
3. **CI/CD連携**: 評価スコアに基づくデプロイゲートが最重要か → ならBraintrust

| 判断軸 | Langfuse | LangSmith | Braintrust | Helicone |
|---|---|---|---|---|
| セルフホスト | ✅ | ❌ | ❌ | △（プロキシ型） |
| 3機能統合 | ✅ | △ | △ | ❌ |
| 導入の速さ | △ | ○ | ○ | ✅（1行変更） |
| コスト管理 | ✅ | ○ | ○ | ✅（キャッシュ機能） |

金融機関や規制業界なら**Langfuse一択**。スタートアップで素早く始めたいならHeliconeから入ってLangfuseに移行するパスもある。

---

### Q2-7. 金融業務でLangfuseを使う場合、PIIマスキング以外に何を気をつけるべき？

**短答**：**3つの追加要件**がある。

1. **監査証跡の保持期間**: 金融規制上、トレースを何年保持する必要があるか。S3のライフサイクルポリシーで対応
2. **アクセス制御**: Langfuse上で誰がどのトレースを見られるかのRBAC。プロジェクト単位でのアクセス分離
3. **ネットワーク分離**: LangfuseをVPC内に閉じ、外部通信を遮断する。第2回で学んだ「分ける」がここでも効く

金融機関向けの構成は、AWS VPC > High Security Zone内にLangfuse（Web/Worker）+ PostgreSQL + ClickHouse + S3 + Redisを配置するのが基本パターン。

---

### Q2-8. LLM-as-a-Judgeの評価指標、Claude Codeの場合は何を設定すべきですか？

**短答**：**Claude Codeの用途に応じて3パターン**ある。

**コード生成の場合**：
- 正確性（テストがパスするか）
- コード品質（命名規則・構造の一貫性）
- セキュリティ（脆弱性パターンの有無）

**ドキュメント生成の場合**：
- 事実の正確性（ハルシネーションの有無）
- フォーマットの遵守
- 過不足のなさ

**RAGパイプラインの場合**：
- 検索精度（関連文書を取得できたか）
- 回答の忠実性（検索結果に基づいているか）
- ハルシネーション率

大事なのは「**レビューの自動化ではなく、品質の変化にデータで気づける仕組み**」として使うこと。スコアが急に下がったときに人間が調べるトリガーにする。

## L3：深掘り質問

### Q3-1. OpenTelemetryネイティブって何が嬉しいんですか？

**短答**：**既存の監視基盤とシームレスに統合できる**。

多くの企業は既にDatadog、Grafana、New Relicなどで監視基盤を持っている。LangfuseがOpenTelemetry（OTel）対応であることで、LLMのトレースデータを**既存のダッシュボードに統合**できる。

具体的なメリット：
- LLMのレイテンシをアプリケーション全体のAPMと同じ画面で見られる
- アラート設定を既存のPagerDuty/Slack連携にそのまま乗せられる
- ベンダーロックインがない ― Langfuseをやめても、OTelデータは他のバックエンドに送れる

LangSmithはLangChain専用、HeliconeはAPI Proxy専用だが、**Langfuseはどのフレームワークからでも標準的なOTelで繋がる**。

---

### Q3-2. Claude Codeのトレース、チームで使う場合はどう運用すべき？

**短答**：**3つの運用ルール**を決めておく。

1. **user_idの命名規則**: 誰のセッションかを一意に特定できるように。GitHub usernameが最も自然
2. **session_idの粒度**: 1つのタスク（1つのPR）= 1 session とするのが実用的。跨ぐとトレースの意味が薄れる
3. **コスト予算のアラート**: user_id × 日次でコスト上限を設定し、超過時にSlack通知

チームのマネージャーは「誰がどのタスクにどれだけコストを使ったか」をLangfuseのダッシュボードで可視化できる。**コスト管理と品質管理が同じ画面で見える**のがLangfuseの強み。

---

### Q3-3. Prompt CMSを本番で使う場合、Claude Codeの開発フローとどう統合する？

**短答**：**第5回で学んだSDDのフローに組み込む**のが自然。

1. **開発時**: Claude CodeでCLAUDE.md + docs/spec.mdに従ってコード生成
2. **LLMアプリのプロンプト**: Prompt CMSで管理。staging環境でテスト → production環境にプロモート
3. **プロンプト変更時**: コードデプロイ不要。ドメイン専門家がPlaygroundでテスト → stagingラベル → 検証OK → productionラベル
4. **問題発生時**: ワンクリックでロールバック。前のバージョンのプロンプトに即座に戻せる

CLAUDE.mdの変更はgit commitが必要だが、**Prompt CMSの変更はgit不要**。この分離がアジリティを劇的に向上させる。

---

### Q3-4. 「トレースなしの運用は計器なしの飛行」と言っていましたが、最小構成は？

**短答**：**最小3ステップで始められる**。

1. **Langfuseアカウント作成**（クラウド版なら5分）or **Docker Composeでセルフホスト**（30分）
2. **SDKをインストール**して、API呼び出しをラップ（コード変更10行程度）
3. **Langfuseダッシュボード**でトレースを確認

評価（LLM-as-a-Judge）やPrompt CMSは後から追加すればいい。**まずはTrace（可視化）だけ始める**のが今日のKey Takeaway。フェーズ1は今日から始められる。

---

### Q3-5. 今日の話はClaude Codeに限った話ですか？他のAIツールにも使えますか？

**短答**：**Langfuseは汎用ツールなので、Claude Code以外にも全て使える**。

- **LLMアプリ**（RAGパイプライン、チャットボット、要約ツール等）→ 本来のメインユースケース
- **Claude Code**（今日のテーマ）→ エージェント型のトレースとして活用
- **他のコーディングAI**（Cursor、Copilot等）→ API呼び出しをラップできれば同様に適用可能

今日はClaude Codeにフォーカスしたが、Langfuseの3機能（Trace / Evaluate / Prompt CMS）の考え方はAIを使うあらゆる場面に共通する。**この勉強会で学んだ「残す」の原則は、ツールが変わっても有効**。

---

### Q3-6. 評価スコアを使って「自動でデプロイを止める」みたいなことはできますか？

**短答**：**Langfuse単体では難しいが、CI/CDと組み合わせれば可能**。Braintrustはこの機能が強みだが、Langfuseでも以下のフローで実現できる：

1. Langfuseの「実験（Experiments）」機能でデータセットに対してバッチテストを実行
2. テスト結果のスコアをAPIで取得
3. GitHub ActionsやCI/CDパイプラインでスコアが閾値を下回ったらデプロイをブロック

ただし現時点では**まずフェーズ1（可視化+評価）を安定させるのが先**。CI/CD連携は次のステップとして検討すべき。

---

### Q3-7. 今日の第6回で「3原則が揃った」とのことですが、後半（第7回〜）との接続はどうなりますか？

**短答**：前半6回で「**土台**」が揃った。後半4回は「**活用を広げる**」フェーズ。

| 回 | テーマ | 前半との接続 |
|---|---|---|
| 第7回 | ユースケース | 第5回のSDDを具体的なワークフローで実践 |
| 第8回 | Agent Teams | 第4回のサブエージェントを組織レベルに拡張 |
| 第9回 | ハーネスエンジニアリング | 第6回のトレースを前提にした自動パイプライン設計 |
| 第10回 | ローカルLLM | 第2回の「分ける」を補完するオフライン戦略 |

特に**第9回「ハーネスエンジニアリング」は今日のLangfuseトレースが前提**になる。パイプラインの各ステップをトレースで可視化・評価できないと、ハーネスの改善サイクルが回らない。
