## S3: なぜ従来の監視では不十分なのか（2-5分）★重要

タイトル: 従来のシステム監視と、生成AIシステム監視の違い
サブタイトル: 生成AI特有の「毎回異なる出力」と「見えない処理過程」が新たな監視のアプローチを必要とする

左カラム：従来のシステム監視（CARD_BG、ネイビー枠）
- ラベル: **予測可能・定型的**
- フロー図: 入力 → 処理 → 出力 ✓
- 焦点: インフラの稼働状況、レイテンシ、エラーレート（HTTP 500など）、データベースのクエリパフォーマンス
- 前提: 「決まった入力に決まった出力が返る」ことが前提
- 手法: ログ、APM、セキュリティ監査 ― いずれも**リソースの状態**を監視する

右カラム：生成AIシステム監視（LIGHT_RED_BG、赤枠で課題を強調）
- ラベル: **予測困難・複雑で不透明**
- フロー図: 入力 → 推論過程(思考) → 検索/ツール/生成 → 反復 → 評価 → 出力 ?
- 3つの新しい監視対象:
  1. **出力の品質と妥当性**: 同じ入力でも回答が変化する。ハルシネーション（事実誤認）や不適切な回答を検知する必要がある
  2. **推論過程の可視化（トレース）**: 複数のステップ（検索、ツール利用、生成）のどこで問題が発生したかを特定するための詳細なログ（トレース）が不可欠
  3. **プロンプトとモデルの挙動**: 時間経過による精度の劣化（ドリフト）や、プロンプトの変更が与える影響を監視・評価する

下部（赤背景で結論）:
> 結論: 「**リソースの監視**」から「**AIの推論過程と出力品質の監視（LLM Observability）**」への拡張が不可欠。
> 従来のDatadog/CloudWatch/New Relicだけではカバーできない領域がある。

## S4: 解決策 ― Langfuse（5-7分）

タイトル: Langfuse ― エンタープライズ品質のオープンソースLLMエンジニアリングプラットフォーム
サブタイトル: AIの思考プロセスを可視化し、安全で信頼性の高い運用を実現するための統合プラットフォーム

中央にLangfuseロゴ的な概念図を配置し、3つの特徴を横3列カードで囲む:

**① 安全な自社環境での運用（セルフホスト）**
- データを外部に出さない安全な運用が可能。オープンソースであり、自社のプライベートな環境（VPC）に構築できる
- **金融機関の厳しいセキュリティ要件**をクリアするためには、セルフホストが事実上の必須要件
- AWS上のセルフホスト構成: PostgreSQL（メタデータ）+ ClickHouse（高速トレース分析）+ S3（大規模ログ保存）+ Redis（キャッシュ・キューイング）

**② 3つのコア機能を統合**
- トレース、評価、プロンプト管理の3つのコア機能を統合
- LLMアプリケーションのライフサイクル全体をカバーする
- 次のスライド（S5）でコアコンセプトを詳しく解説

**③ OpenTelemetryネイティブ**
- 業界標準の規格に対応しており、特定のベンダーに依存しない
- すでにお使いの監視ツール（Datadog、Grafana等）とも簡単に連携できる
- LangChain、LlamaIndex、OpenAI SDKなど主要フレームワークとの統合が標準装備

下部:
> Langfuseは単なるログ収集ツールではなく、見え辛いブラックボックス化されたAIエージェントを
> 「**監査可能**」かつ「**継続的に最適化可能**」なエンタープライズ資産に変えるインフラ。

## S5: コアコンセプト ― Trace → Evaluate → Improve（7-9分）★核心

タイトル: 推測を排除し「データに基づくLLMOpsサイクル」を回す
サブタイトル: AIエージェントの運用プロセスを「可視化」「評価」「改善」のシームレスなループへと根本から変革する

3ステップのフロー図（横3列、大きな矢印で接続。各ステップに番号①②③を付与）:

**① 可視化（Trace）: 振る舞いの解明**
- すべてのLLM呼び出し、検索（RAG）、API実行のステップを**階層的に記録**
- コストと遅延を**ユーザー単位**でリアルタイムに追跡
- 「何が起きているか」を把握できる状態にする

**→（矢印）**

**② 評価（Evaluate）: 品質劣化の検知**
- LLM-as-a-Judge（AIによる自動評価）やユーザーフィードバックを活用
- 非決定的な出力の品質（正確性、関連性など）を**定量的なスコア**としてトレースに紐付け
- 「品質がどの程度か」を数値で語れる状態にする

**→（矢印）**

**③ 改善（Prompt CMS）: 迅速な修正とデプロイ**
- 蓄積されたデータを基にプロンプトを改修
- ソースコードからプロンプトを切り離し、UI上から安全にバージョン管理と本番反映を行う
- 「修正→テスト→デプロイ」を**コード変更なし**で回せる状態にする

下部（ネイビー背景の帯）:
> 結果: 障害対応時間の劇的な短縮と、**コンプライアンス要件に応える完全な監査ログ**の確立。
> このサイクルが「残す」の実装であり、第5回の「計画→実装→記録」ループを**システムレベルで自動化**したもの。

    ├─ Planning Step     1.2s  $0.02  740 tokens

## S7: Claude Code のトレース ― なぜ開発ツールにもObservabilityが要るのか（10-11分）

タイトル: Claude Code のトレースを取る意味
サブタイトル: 「コードを書くエージェント」の振る舞いこそ、可視化しないと改善できない

左カラム：Claude Code特有の課題（LIGHT_RED_BG、赤枠）
- Claude Codeは**ターミナルで直接ファイルを読み書きし、コマンドを実行する**エージェント（第1回の復習）
- 1つの指示に対して内部で複数のステップ（ファイル読み込み → 計画 → コード生成 → テスト実行 → 修正）を**自律的に反復**する
- 問題: 「指示通りに動かなかった」とき、**どのステップで判断を間違えたのかがターミナルログだけでは分からない**
- 典型的なトラブル:
  - 間違ったファイルを読んで前提を誤った（コンテキスト汚染）
  - 不要なツールを呼んでトークンを浪費した（コスト爆発）
  - 生成→テスト→修正のループが無限に回った（無限ループ）
  - CLAUDE.mdの指示を途中で忘れた（コンテキストウィンドウ溢れ）

右カラム：Langfuseで実現できること（CARD_BG、ネイビー枠）
- **Claude Code → Langfuse連携**により、エージェントの全ステップをトレースとして記録できる
- Claude CodeのAPI呼び出し（Anthropic Messages API）をLangfuseのSDKでラップするだけで、以下が自動記録される:
  - 各ステップの入力プロンプト・出力・トークン数・コスト・レイテンシ
  - ツール使用の履歴（どのファイルを読んだか、どのコマンドを実行したか）
  - セッション単位での累計コスト
- トレースツリーのイメージ（コードブロック、ダークネイビー背景 #0F1434）:
  ```
  Claude Code Session: "認証機能を実装して"
    ├─ Read: src/auth/login.ts          0.3s   120 tokens
    ├─ Read: CLAUDE.md                   0.1s    80 tokens
    ├─ Planning                          1.5s   950 tokens  $0.03
    ├─ Edit: src/auth/login.ts           0.8s   600 tokens  $0.02
    ├─ Bash: npm run test               3.2s     0 tokens
    ├─ Read: test output (FAIL)          0.1s   200 tokens
    ├─ Planning (retry)                  1.2s   800 tokens  $0.02
    ├─ Edit: src/auth/login.ts           0.6s   450 tokens  $0.01
    ├─ Bash: npm run test               2.8s     0 tokens
    └─ Read: test output (PASS)          0.1s   150 tokens
  Total: 10.7s  3,350 tokens  $0.08
  ```

下部（赤背景帯で強調）:
> **第3回で学んだHooksと組み合わせる**と、Hooksでブロックしたコマンドの記録もトレースに残る。
> 「防ぐ（Hooks）」と「残す（トレース）」が**連動する**ことで、セキュリティと監査の両方をカバーする。
> Claude Codeを業務で使うなら、**トレースなしの運用は計器なしの飛行と同じ**。

## S8: 機能② 評価 ― LLM-as-a-Judge（11-12分）★重要

タイトル: 機能詳細②: 評価（Evaluation / LLM-as-a-Judge）
サブタイトル: 出力が毎回変動する生成AIシステムにおいて、スケーラブルかつ客観的な品質保証（QA）を実現する

左カラム：LLM-as-a-Judgeによる自動スコアリング
- 高性能なLLM（GPT-4等）を「審査員」として定義し、本番環境のトレースに対して自動でスコアリングを実行
- 評価指標の例: 正確性（Correctness）、関連性（Relevance）、ハルシネーションの有無、ブランドボイスの遵守など
- **金融業務の例**: 市況コメンタリーの事実関係チェック、規制用語の正確な使用、顧客向け回答の適切性

中央カラム：多様な評価手法の統合
- 自動評価だけでなく、**エンドユーザーからのフィードバック**（👍👎）や、**社内ドメイン専門家による手動アノテーション**を統合
- スコアは「数値」「ブール値（True/False）」「カテゴリ値」として元のトレースデータに直接紐付けられる
- 3つの評価ソースを統合することで、単一指標では見落とす品質問題を網羅的にカバー

右カラム：RAGパイプラインの診断
- Ragasなどの外部評価フレームワークと統合し、コンテキストの関連性や回答の忠実性を定量化
- **検索ステップの精度不足か、生成ステップのハルシネーションかを切り分ける** ← これが従来の監視では不可能だった部分
- 金融業務クエリに対して「検索した社内文書が古かった」のか「正しい文書から誤った回答を生成した」のかを特定できる

下部:
> スコアがトレースに紐付くから、「**いつ・どこで・なぜ品質が下がったか**」をデータで語れる。
> 目視確認では月に数件しかチェックできない → LLM-as-a-Judgeなら**全件自動チェック**が可能。

## S9: 機能③ Prompt CMS（12-13分）

タイトル: 機能詳細③: プロンプト管理（Prompt CMS）
サブタイトル: プロンプトを「ハードコードされた文字列」から「バージョン管理可能な独立したアセット」へと昇華させる

左カラム：コードベースからの完全な分離
- プロンプトの微調整を行うたびに発生していた「**エンジニアによるコード変更・再デプロイ**」のサイクルを排除
- API経由で最新のプロンプトをミリ秒単位の低遅延で取得（Redisキャッシュを活用）
- 現状の課題: プロンプトの修正が依然としてコードデプロイに依存しており、**アジリティに制限**が生じている

中央カラム：安全なデプロイメントとバージョン管理
- すべての変更は新しいバージョンとして記録され、即座なロールバック（巻き戻し）が可能
- production（本番用）、staging（検証用）といった**ラベル機能**を活用し、安全なリリース管理を実現
- コードブロック例（ダークネイビー背景 #0F1434）:
  ```
  PROMPT:
  "You are a helpful assistant
   for financial analysis."
  production → v2.0
  staging    → v2.1
  ↩ Rollback Capability
  ```

右カラム：ドメイン専門家とのコラボレーション
- 非技術者（**金融業務の専門家やコンプライアンス担当者**）が直接LangfuseのUI上でプロンプトをテストし、改善・デプロイできる環境を提供
- Playground機能でプロンプトを試行 → 結果を比較 → ラベルを付与して本番反映
- 開発サイクルを大幅に加速: 日分からのプロンプト調整サイクルを**分単位**に短縮

下部:
> プロンプト修正がコードデプロイに依存しなくなる = **デプロイサイクルの分離によるアジリティの劇的な向上**。
> これは第5回で学んだSDDの「ステアリングファイル」をシステムレベルに拡張した概念。
