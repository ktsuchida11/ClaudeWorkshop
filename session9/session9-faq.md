# 第9回 開発へのループ応用 ― 想定 Q&A

> **核フレーズ**：「実装と検証を ISSUE で繋ぐ」
> **位置づけ**：第8回で確立したハーネスの「型」を、ソフトウェア開発のループへ移し替える回。実装役と検証役を物理的に分け、やりとりを ISSUE テキストに残して往復するのが要。
> **3層構成**：L1（未経験者・全体像）/ L2（日常利用者・実装と運用）/ L3（深掘り・設計判断と自律度）。

---

## L1 ― 未経験者向け（全体像・用語）

### Q1-1　「ループ」とは？「ハーネス」とどう違う？
**A.** **ハーネス＝エージェント1体分の装備（1回を賢くする足場）**、**ループ＝自走の仕組み（止め方込みで回し続ける）**です。ループはハーネスを内側に含みます。第8回のエージェントは名前は「ハーネス」でも、定時起動＋自己反省＋日次改善で実体はループ。両者の違いは「**止め方と評価を設計しているか**」にあります。

### Q1-2　なぜコード開発は「ループに向く」の？
**A.** コードは**客観評価を自動化しやすい**からです。テスト・ビルド・型・lint が機械的に YES/NO を返すので、レポート生成（筆者の主観 FB に依存）よりむしろループ向き。鍵は「**テストと受け入れ条件を gold_standard（正解）に据える**」ことです。

### Q1-3　レポート生成と開発工程で、ループの「何が変わる」？
**A.** 変わるのは**改善シグナルの出どころだけ**で、二重ループ（実装ループ＋学習ループ）の構造は同じです。レポートでは改善を**筆者の主観 FB（人手・不定期）**が握っていましたが、開発では**テスト結果・受け入れ条件（客観・自動）**が毎回返ります。表の列を「レポート生成 → 開発工程」に読み替え、正解＝テスト、接点＝PR に差し替えればそのまま乗ります。

## L2 ― 日常利用者向け（実装・運用）

### Q2-1　アプリ開発でこのループを始めるには、まず何から？
**A.** **評価基準＝テスト／受け入れ条件を先に書く**ことです。これが gold_standard になり、無ければループは回りません。次に CI で自動・定期に走らせて差分が毎回届く状態にし、生成コードは“動かして”検証（build→型→lint→test→e2e）。人間は PR レビューで方向づけ・最終マージ。最初は1リポ／1機能で baseline を取り、規約は増やしすぎず小さく回します。

### Q2-2　「実装と検証を分ける」とは具体的にどう分ける？ ISSUE テキストの役割は？（核心）
**A.** 実装役（Generator）と検証役（Evaluator）を**別セッション／別 worktree** に物理的に分け、**両者のやりとりを必ず ISSUE テキストに残して往復**させます。ISSUE には受け入れ条件・再現手順・期待結果＝gold_standard を記述し、検証結果・指摘も ISSUE に追記。口頭で渡さず文章に残すことで、**非同期・記録・責務分離**が成立します。実装役は ISSUE から正解を読むだけで検証結果は見ない（採点者に最適化しない）、検証役は合否と理由を ISSUE に書き戻すが実装は直さない、という分担です。

### Q2-3　検証ゲートとは？ /goal とは？
**A.** **丸投げとループの分水嶺は“検証”**です。「生成（見た目）」ではなく「**動作**」を合格条件にする ―― build→型→lint→test→e2e で実際に動かして初めて通す。`/goal` は「テスト100個を全部通す」「速度を50%改善」のように、**合格まで AI が回り続けるループの最も使いやすい入口**です。正解がふわっとした仕事は空回りするので、受け入れ条件＝gold_standard を先に書くことに接続します。

### Q2-4　どんなタスクがループに向く／向かない？（設計ステップ0）
**A.** **向く**＝独立した成果物が複数（並列可）・合否を機械的に言語化できる・繰り返し発生する・入出力と依存が明確。**向かない**＝探索中／一回きり／軽い修正／品質が主観的でよい／反復コストが効果を上回る。判断軸は「**何度繰り返すか**」と「**合否を言語化できるか**」。「軽い修正」と思って手で始めると後悔しがちなので、まず“ループ化すべきか”を判定します。

### Q2-5　並列生成や役割分担（PGE）はどう実装する？
**A.** Planner-Generator-Evaluator を**別コンテキスト**に分け、基準を満たすまで2〜3周します。Generator は **worktree で分担**してコンフリクトを回避し並列生成、Evaluator は別セッションでテスト／静的解析／レビュー観点で採点。**フィードバックループの有無が品質の分水嶺**（1回だと50〜60%で人手に回りレビュー負荷増、2〜3周で70〜80%）。モデル使い分け（設計は高性能・実装は中位・検査は軽量）の本質は「**チェック基準を言語化できるほど安いモデルに置換できる**」ことです。

## L3 ― 深掘り（設計判断・自律度）

### Q3-1　概念マッピング：第8回の各要素はコードで何に当たる？
**A.** 対応は次のとおり ―― **gold_standard ＝ テストスイート＋受け入れ条件**（＋あれば参照実装）／**S1・S2・S3 ルーブリック ＝ テスト合格率・型/lint・ビルド成否・ベンチ・レビュー観点（正確性/可読性/安全性）**／**評価情報の定期受領 ＝ CI 結果・PR レビュー・本番テレメトリ/エラーログ**／**Fact-Check ＝ 静的解析/型/lint（内部）＋実行・結合・e2e（外部＝実際に動かす）**／**style_guide ＝ コーディング規約・「二度とやらない」教訓**／**メール返信 FB ＝ PR レビューコメント**／**自動送信禁止 ＝ 自動マージ／デプロイ禁止**。締めの一言は「テストが gold_standard、CI が教師データ配信、PR レビューがメール FB」。

### Q3-2　どう設計すると開発が「人の手を離れる」？（自律度）
**A.** 人間のチェックポイントを Lv0→Lv4 と**外側へ**動かします。各段の**通行証は「評価の自動化」** ―― 人間の判断を機械や別エージェントが代行できた時だけ次の段へ進めます。だから**評価基準の言語化が全段の律速**。最後まで残る人間の仕事は「**尋問**」（「本当に必要？」「どっちが上位概念？」＝実装を理解せず出せる問い）で、これを外部レビュアに移すのが Lv3→4。完全には消えず「1日1分の最終承認」に収束します。検証・理解・判断は手放さないのが原則。

### Q3-3　暴走（コスト爆発）を防ぐには？「意図の負債」とは？
**A.** **「止め方を決めていないループは、ただの請求書」**。停止条件を3つ必ず入れます ―― ①回数上限 ②進捗チェック（数回変化なしなら潔く止める）③予算上限（トークン／金額で強制終了）。もう一つの落とし穴が**意図の負債（Intention Debt）**＝差分を読まずにマージし続けると、誰も中身を知らないコードが積み上がること。生成が速いほど効くので、「**差分を毎回読む**」ことだけは手放しません。

### Q3-4　二重ループ（実装ループ＋学習ループ）とは？
**A.** どんなループも「**速く作る（実装ループ）**」と「**人の介在を減らす＝自律度を上げる（学習ループ）**」の二重構造です。第8回のエージェントはこの両方を既に備えた縮図で、開発に移すときは**正解＝テスト、接点＝PR に差し替えるだけ**。実装ループは生成→検証(build/型/test/e2e)→PR、学習ループはレビュー指摘→規約・教訓の蓄積、という対応になります。
