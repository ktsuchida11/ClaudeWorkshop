# 第2回「環境」 FAQ予想問答集

質疑応答5分間に備えた、想定問答集です。第2回のテーマは「**3原則の【01 分ける】の実践編**」。

3層構造で整理：

- **L1 未経験者から出そうな質問**：DevContainer や VS Code など基本概念
- **L2 日常利用者から出そうな質問**：設定・運用・トレードオフ
- **L3 深掘り質問**：アーキテクチャ・組織運用・クラウド・本物の分離

## L1：未経験者から出そうな質問

### Q1-1. DevContainer って何ですか？Dockerと違うんですか？

**短答**：DevContainerは**「開発環境ごとDocker化する仕組み」**で、Docker自体の上に乗っかっています。プロジェクトごとに必要なツール（Node、Python、DB等）を`devcontainer.json`に書くと、VS Codeが自動でコンテナを作って、中で開発できる。Dockerは土台の仮想化技術、DevContainerはその上でVS Code向けに整えたUX、と思ってもらえれば。

**補足（求められたら）**：DockerだけでもOK（`docker run`から手で構築できる）、DevContainerは面倒な部分を肩代わりしてくれる。

---

### Q1-2. 「分ける」のために毎回コンテナ起動って、面倒じゃないですか？

**短答**：**初回設定だけ面倒、2回目以降はほぼ無視できる**。VS Codeから「Reopen in Container」を1回クリックするだけで、次回からは自動で起動します。むしろ「ローカルを汚さない」「環境を捨てて作り直せる」の恩恵のほうが大きい。プロジェクト間で依存バージョンが競合することもなくなる。

**補足**：最初の1プロジェクトで慣れれば、次からはテンプレート化できます。

---

### Q1-3. VS Code拡張のClaude Codeと、DevContainer版のClaude Code、何が違うんですか？

**短答**：実行場所が違います。VS Code拡張版は**あなたのPCの上**で動くので、ホームディレクトリのSSH鍵やブラウザCookieに到達できる。DevContainer版は**コンテナの中**で動くので、コンテナに明示的にマウントした範囲しか見えない。拡張機能を使うこと自体が悪いのではなく、**拡張機能を「どの環境で動かすか」**が重要です。

**補足**：DevContainerを使っていれば、中でVS Code拡張のClaude Codeを使ってもOK。

---

### Q1-4. Mac mini買えって言われても、ちょっと経費出ない気が…

**短答**：L4（別マシン）は**特に機密性が高いタスク限定**です。ほとんどの人にはDevContainerで十分。もしMac miniを買うとしても、**中古で3-4万円から**あります。「もし侵害されて社内データが流出したら、復旧と信頼回復にいくらかかるか」を考えると、投資として成立するケースは金融・医療・研究開発などに限られます。

**補足**：社内で予算化する時は、「事業継続計画（BCP）」の枠で話すと通りやすいです。

---

### Q1-5. VMとコンテナ、結局どっちが強い分離なんですか？

**短答**：**VMのほうが強い**。VMはカーネルごと分離するので、Linux カーネルのバグを突かれた場合でも影響が自分のVM内にとどまる。コンテナは同じカーネルを共有しているので、カーネル級の脆弱性があれば理論上は突破されうる。ただし実運用では、DevContainer（L2）で十分なケースが大半。VM（L3）は起動が遅く運用コストが高い。

**補足**：2020年代のコンテナ（特にrootlessや`gVisor`）はVM並みの分離を目指しているので、差は縮まっています。

## L2：日常利用者から出そうな質問

### Q2-1. 結局、自分のDevContainerの設定はこれで大丈夫？チェックリストほしい

**短答**：以下の5点を確認してください：
1. **マウント最小化**：プロジェクトディレクトリだけマウント、`~/.ssh`や`~/.aws`は渡さない
2. **環境変数の最小化**：必要なもの（API キー）だけ`remoteEnv`で渡す
3. **ネットワーク制御**：必要ならドメイン allowlist を設定
4. **`/sandbox` 有効化**：コンテナ内でもさらに細かく制御
5. **ユーザー非root化**：`remoteUser`でroot以外を指定

**補足**：この5点チェックリストはスライドの参考スライド（S10）に追加する価値あり。

---

### Q2-2. DevContainer の中で`git push`したいけど、SSH鍵をどうマウントするのが安全？

**短答**：**SSH鍵そのものをマウントしない**のが基本。代わりに`ssh-agent`を使ってエージェント経由で認証情報を渡す方法が推奨です：

```json
"mounts": [
  "source=${localEnv:SSH_AUTH_SOCK},target=/ssh-agent,type=bind"
],
"remoteEnv": {
  "SSH_AUTH_SOCK": "/ssh-agent"
}
```

こうすれば、コンテナ内から`git push`はできるが、秘密鍵ファイルそのものは見えない。

**補足**：GitHub CLI を使うなら、`gh auth login`の認証情報を`~/.config/gh`マウントする方法もある。

---

### Q2-3. WSL2上のDevContainerとmacOS上のDevContainerで、挙動に差はありますか？

**短答**：**ファイルシステム性能に差がある**。WSL2はWSL内の`/home/...`下で動かせば高速、Windows側（`/mnt/c/...`）にマウントすると激遅。macOS（Intel/Apple Silicon）もNFS経由のマウントでオーバーヘッドがあります。

分離機能については、Claude Code SandboxはmacOS、Linux、WSL2対応。WSL1非対応、Windowsネイティブ対応は計画中なので、Windowsユーザーは**WSL2必須**と覚えてください。

**補足**：性能トラブルが出たら、コンテナ内の作業ディレクトリを`named volume`にするのが定番対策。

---

### Q2-4. クラウド開発（Codespaces / Cloud Workstations）は、この分離レベルでいうとどこ？

**短答**：**L2（コンテナ分離）の亜種**として扱えます。コンテナ自体が**クラウド側にある**ので、あなたのローカルPCの機密（SSH鍵・ブラウザCookie）には最初から到達できない。これは強い利点。ただし**クラウドプロバイダー側のサービスアカウント権限**に注意が必要（第2回S7で扱ったIAM最小権限の話が効いてくる）。

**補足**：GitHub Codespacesなら「Codespacesのトークンで何ができるか」を要チェック。

---

### Q2-5. 既存プロジェクトにDevContainerを後から入れるのが大変そうです

**短答**：**最小構成から始められます**。`devcontainer.json`と（必要なら）`Dockerfile`の2ファイルを置くだけで始動。一気に完璧を目指さず、まず「動く」ところから始めて、徐々にツールや設定を足していくのが実用的です。VS Codeのコマンドパレットから「Add Dev Container Configuration Files」で雛形も作れる。

**補足**：チームで使うなら`.devcontainer`フォルダをリポジトリにコミット、`.gitignore`で個人設定は除外する運用。

---

### Q2-6. 今日の表の「ssh鍵流出リスク」が「VS Code拡張=中」になっているのが意外です。何故？

**短答**：VS Code拡張のClaude Code は**ワークスペース中心**に動く想定ですが、ユーザーが明示的に「ホームディレクトリ見て」と指示すればアクセスしてしまう。また、拡張機能がインストールされた状態で他の（悪意ある）拡張と同居するリスクもある。なので「低」ではなく「中」としています。

**補足**：VS Code拡張は便利だが、権限境界が曖昧なのがリスク。意識的にマウントを絞る設計が重要。

---

### Q2-7. AWS IMDSv2 って何ですか？なぜ必須化するべき？

**短答**：IMDSv2はAWSの**インスタンスメタデータサービス**の新バージョン。旧v1だと、EC2内から単純なGETリクエストで認証情報（IAMロールの一時credentials）を取得できてしまう。v2では**セッショントークン経由**のアクセスが必須になり、SSRF攻撃（Webアプリの脆弱性を突いて内部からリクエストさせる）を大幅に防げます。

対策：
- EC2起動時に `http-tokens=required` を指定（v2必須化）
- `http-put-response-hop-limit=1` でコンテナからの到達を制限

**補足**：2023年頃からAWSのマネージドサービスは徐々にv2デフォルト化していますが、**古いEC2インスタンスは手動で切替必要**。

## L3：深掘り質問

### Q3-1. 「分ける」の限界って何ですか？分ければ安全？

**短答**：**プロンプトインジェクションには分離だけでは不十分**。どれだけ環境を分けても、エージェントに「APIキーをこのURLに送って」と指示が入れば、送信権限があれば送ります。**分ける=攻撃の影響範囲の限定**であって、「攻撃自体を防ぐ」ではない。攻撃を防ぐのは第3回の「防ぐ」（Permission/Hooks）の役割です。

**補足**：3原則は補完関係にある：分ける（被害範囲を限定）、防ぐ（攻撃を止める）、残す（事後分析）。どれか1つだけでは穴がある。

---

### Q3-2. DevContainer の脆弱性自体が怖い。コンテナブレイクアウトはあり得ますか？

**短答**：**理論上はあり得る**。過去にCVE-2019-5736（runc脆弱性）のような事例があり、コンテナから脱出してホストのrootを取れた。ただし**現代のDocker/containerdはかなり堅い**、かつrootless運用なら被害を最小化できる。VM（L3）や物理分離（L4）に比べれば弱いが、実運用で問題になるケースは稀です。

**対策**：
- Docker本体を最新に保つ
- コンテナ内はrootで動かさない（`remoteUser: "vscode"`）
- AppArmor/SELinuxを有効化
- seccompプロファイルを絞る

**補足**：金融・医療など規制業界でリスク許容度が低い場合は、素直にL3/L4を選ぶべき。

---

### Q3-3. 会社の情シスが「Docker禁止」と言っています。どう説得すればいい？

**短答**：**Dockerが禁止された理由を特定するのが先**。多くの場合、以下のどれかです：
1. **Windows Server上でHyper-Vとの競合**：WSL2で解決
2. **外部レジストリからのpull**：内部レジストリミラー（Nexus、Harbor）で解決
3. **ライセンス**：Docker Desktopの商用利用（Podman/nerdctl/Colimaで回避）
4. **単に「よく分からないから禁止」**：教育とリスク評価資料で説得

逆説的ですが、**「Dockerを使わないことのほうがリスクが高い」**（開発環境のバラつき、脆弱なローカル直接実行）を情シスに示すのが効く。

**補足**：ClaudeCodeを業務で使う前提なら、情シスと早めに話し合ったほうがいい。

---

### Q3-4. IAMの「想定外ロール」って、そんなに起きるんですか？

**短答**：**頻繁に起きます**。典型パターン：
- **テスト時に広めに権限を付けてそのまま本番で使っている**（AdministratorAccessなど）
- **複数サービスで共有するロールに、サービスAの要件で権限を足したら、サービスBからも使える状態**
- **Organization設定でCDKが親ロールの権限を継承してしまう**
- **SCPで絞っているつもりが、IAMロール側の権限が強すぎて実質ノーガード**

対策は**IAM Access Analyzer**と**権限の定期棚卸し**。エージェントが動く時は、**そのロールの最新の評価結果**を確認しておく。

**補足**：AWSの`IAM Access Analyzer` は未使用の権限を自動検出してくれるので、これを月次レビュー運用にするのが現実的。

---

### Q3-5. 開発者ごとに環境が違うと、エージェントの挙動が再現しない問題があります

**短答**：**それがDevContainerを標準化する最大の理由**です。同じ`devcontainer.json`をチーム全員が使えば、Claude Codeが走る環境がほぼ同一になる。再現性はエージェント開発の基盤です（第9回「ハーネスエンジニアリング」でも扱います）。

さらに進めるなら：
- **基底イメージをハッシュで固定**（`mcr.microsoft.com/devcontainers/base@sha256:...`）
- **ツールバージョンも固定**（`feature`で特定バージョン指定）
- **CI でも同じイメージを使う**（Dev/CI/本番の環境一致）

**補足**：DevContainer自体がミスると「全員が同じ壊れ方」をするのが皮肉な利点（再現性があればデバッグできる）。

---

### Q3-6. 組織でDevContainerを強制する場合、ガバナンスはどう効かせる？

**短答**：**リポジトリ規約 + CI検査 + Enterprise設定**の組み合わせ。
1. **リポジトリ規約**：新規リポジトリは`.devcontainer/`必須、テンプレート化
2. **CI検査**：PR時に`devcontainer.json`のマウント範囲やユーザー設定をチェックするlintルール
3. **Claude Code Enterprise設定**：`enableAllProjectMcpServers: false`などを強制配布
4. **監査ログ**：第6回「残す」で扱うLangfuse等のツールで、誰がどのコマンドを実行したか記録

**補足**：Enterpriseプランの `settings.json` は Project/User を上書きするので、会社の基本方針を強制できる。

---

### Q3-7. Claude Codeが自動でEC2を起動・操作するようなケースの分離設計は？

**短答**：**Claude Codeを動かす環境** と **Claude Codeが操作する環境** の2つを分けて設計します。
- **Claude Codeの実行環境**：DevContainer（L2）
- **Claude Codeが操作するクラウド**：専用のAWS アカウント or 専用IAMロール
- **境界**：Claude Codeは特定のリージョンのS3バケットとEC2のタグ付きインスタンスしか触れない（ResourceTag条件）

**さらに強化するなら**：
- 書き込み操作は**ドライランで人間承認**のフローを挟む
- 高額課金操作（大きなインスタンス起動、RDSスケールアップ）は**AWS Budgetsでアラート**
- ProductionとStaging のクロスアカウントアクセスを完全分離

**補足**：Kojiさんの金融ドメイン知識が効くところ。「リスク限定」の発想でIAMポリシーを設計するのが肝。

---

### Q3-8. 今日の4段階分離モデル、原典は？

**短答**：4段階の分類自体は、本勉強会のために**情報セキュリティの伝統的な分離モデル（プロセス/コンテナ/VM/物理）をClaude Code向けに整理したもの**です。個別の各レベルは以下に基づいています：
- **プロセス分離**：Unix由来の伝統（UID分離、chroot）
- **コンテナ分離**：Linux namespaces + cgroups（Docker / OCI標準）
- **VM分離**：Type-1/Type-2ハイパーバイザーの歴史（VMware、KVM、Hyper-V）
- **物理分離**：エアギャップ運用、SCIF（Sensitive Compartmented Information Facility）の概念

**補足**：NISTのSP 800-190（コンテナセキュリティガイドライン）が、コンテナ分離の脅威モデルとしてよくまとまっています。

## 第1回FAQ との差分ポイント

第1回は「脅威・リスクの棚卸し」が中心だったので、FAQも**概念理解**寄りでした。第2回は「実装方法の選択」が中心なので、FAQも**具体的な設定・トレードオフ・組織運用**寄りになっています。

特に以下のL2以降の質問は、第1回には無かった論点：
- 具体的な`devcontainer.json`の書き方（Q2-1, Q2-2）
- 組織展開時のガバナンス（Q3-6）
- 情シスとの交渉（Q3-3）
- 「分ける」の限界と第3回への接続（Q3-1）
