# 第7回「ユースケース」 FAQ予想問答集

質疑応答5分間に備えた想定問答集。第7回のテーマは「**Claude Codeはコードを書く道具ではなく、仕事の型を実装する道具**」。

3層構造で整理：

- **L1 未経験者から出そうな質問**：3つのレイヤー・Plan Mode・基本的なユースケースの理解（6問）
- **L2 日常利用者から出そうな質問**：Skill Creator・3つの主要型・データ分析・Routinesの実践（9問 ※スライド構成変更でQ2-0追加）
- **L3 深掘り質問**：SubAgentsの設計・Skillの運用・組織展開（6問+メタ戦術）

---

## L1：未経験者から出そうな質問

### Q1-1. 「3つのレイヤー」って言ってましたが、最初はレイヤー1だけでいいですか？

**短答**：**はい、レイヤー1から始めるのが正解**。対話型で「バグを直して」「テストを書いて」とClaude Codeに直接頼むだけで、十分に生産性は上がる。

レイヤー2（Skills/コマンド）に進むタイミングは「**同じ指示を3回以上繰り返したとき**」。そのタイミングでSkillに切り出せば、4回目以降は `/skill-name` 一発で済む。レイヤー3（Routines）は「人間がいなくても動いてほしい定型作業」が出てきたら検討する。焦って全部やる必要はない。

---

### Q1-2. Plan Modeって何ですか？普通のモードと何が違う？

**短答**：**Plan Modeはコードを変更しない「読み取り専用」の分析モード**。

通常モードではClaude Codeがファイルを編集したりコマンドを実行したりするが、Plan Modeでは分析と計画だけを行う。`Shift+Tab` を2回押すとONになる。使いどころは3つ：

1. **新しいコードベースの理解**：全体像を掴むときに安全に調査できる
2. **大規模リファクタの事前調査**：影響範囲を特定してから実装に移れる
3. **移行計画の策定**：「OAuth2に移行する計画を作って」のように、実行前に計画を立てる

計画ができたら通常モードに戻して実装させる、という2段階の使い方が効果的。

---

### Q1-3. データ分析ってExcelでやるのと何が違うの？

**短答**：**Claude Codeはプログラミングの力でExcelの限界を超える**。

Excelでは難しいことがClaude Codeでは自然言語で指示するだけでできる：
- **複数ファイルのマージ**: 「この3つのCSVをcustomer_idで結合して」
- **複雑な集計**: pandasで前年同期比・移動平均・ボラティリティ計算
- **可視化**: matplotlib/plotlyでグラフ生成
- **パイプライン化**: 一回限りの分析を再利用可能なスクリプトに変換

ただし**Claude Codeはターミナルで動く**ので、Excelのように「セルを見ながら対話的に操作」はできない。Jupyter Notebookと並行作業するのが実用的。

---

### Q1-4. /rewindって何ですか？間違えても戻せるの？

**短答**：**はい、/rewindはClaude Codeの「元に戻す」機能**。

`/rewind` を実行すると、会話履歴とコードの状態を過去のチェックポイントに巻き戻せる。Claude Codeが間違った修正をした場合でも、修正前の状態に安全に戻れる。

- **Esc**: Claude Codeを途中で止める（コンテキストは保持）
- **Esc → Esc（2回）** or **/rewind**: 巻き戻しメニューを開く
- 会話だけ戻す / コードだけ戻す / 両方戻す を選択可能

「試行錯誤が怖い」という心理的ハードルを/rewindが解消してくれる。

---

### Q1-5. Skill Creatorって、プログラミング知識がなくても使えますか？

**短答**：**使える**。Skill Creatorは対話形式で進むので、プログラミング知識は不要。

`/skill-creator` を実行すると、Claude Codeが「何をするSkillですか？」「いつ発動すべきですか？」と質問してくる。答えていくだけでSKILL.mdが自動生成される。テストも自動で作成・実行される。

Skill Creatorの公式ドキュメントにも、技術者以外のユーザーも想定していると記載がある。**対話に答えるだけでSkillが完成する**のがSkill Creatorの価値。

---

### Q1-6. Routinesって無料で使えますか？

**短答**：**Routinesは研究プレビュー段階**。現時点ではClaude Codeのサブスクリプション（ProまたはMaxプラン）が必要。

RoutinesはAnthropicのクラウド上で実行されるため、ローカルのClaude Code CLIとは別の仕組み。claude.ai/code/routinesからWebで作成できる。実行回数には日次の制限がある。将来的な料金体系は変更される可能性がある。

## L2：日常利用者から出そうな質問

### Q2-0. なぜ Skill の型を「8つ」ではなく「3つ」に絞って紹介したんですか？

**短答**：**「8つ全部覚えよう」とすると、結局どれも使わずに終わるから**。30分の本編で8型のテーブルを見せても、参加者は型の名前を覚えるだけで疲れて、Skill を1つも作らずに帰る ― これが過去版で起きていた問題でした。

絞った3型（Reference / Task / Document）の選定基準は次の3つ：

1. **発動の頻度が高い**：日常業務で繰り返し使うパターンを最初に押さえる
2. **公式 anthropics/skills でサンプルが揃っている**：自作する前に動くものを試せる
3. **3つを使い分けるだけで「型を資産化する」が成立する**：Reference で規約、Task で繰り返し、Document で文書

**残り5型は「困ったら参照する」で十分**。Visual Output が要る場面（コードベース可視化など）は具体的ニーズが先に来るので、そのとき GitHub anthropics/skills を見れば事例が出てきます。**「型を全部覚える」ではなく「必要になったとき型から選ぶ」**がこのスライドの設計意図です。

**補足**：このスタンスは shadcn/ui のドキュメント設計（「全部入りではなく、必要なものをコピペ」）から着想しています。

---

### Q2-1. 今日紹介した「3つの主要型」、実際にはどの型から作り始めるべき？

**短答**：**① Reference型 と ② Task型 の2つから**。残り（Document含む）は必要になってから。

- **① Reference型**: チームのコーディング規約・レビュー基準をSkillにする。CLAUDE.mdが長くなりすぎた部分を切り出すのに最適
- **② Task型**: デプロイ手順・コミットルールなど、毎回同じ手順を踏む作業をSkillにする
- **③ Document型**: docx/pdf/pptx/xlsx を扱う必要が出たら anthropics/skills を `/plugin install` するだけ。自作は不要

最初は **① と ② の2つで「毎回同じことを説明する／同じ手順を実行する」最大の時間浪費を解消**できる。Document型は「今日 docx 触る予定がある」人だけ先に入れておくと良い。

**他の5型（Visual Output / Dynamic Context / Forked Subagent / Background Knowledge / Meta Skill）** は本編で深追いしないと言いましたが、用途は GitHub anthropics/skills の各サンプルディレクトリで確認できます。実際に「コードベース構造を可視化したい」「PR情報を Skill に動的注入したい」など具体的ニーズが出たときに、必要な型を選んで参照する形でOKです。

---

### Q2-2. Skill Creatorの「Skillあり/なし比較」って、具体的にどう見るの？

**短答**：**ワークスペース内のwith_skill/ と without_skill/ ディレクトリの出力を比較する**。

Skill Creatorが自動でSubAgentを2つ起動し、同じプロンプトに対して「Skillあり」と「Skillなし」で並列実行する。出力ファイルを見比べて「Skillがあると品質が上がっているか」を判断する。定量的な評価（assertions）も設定できるが、最初は目視比較で十分。

---

### Q2-3. anthropics/skills のPluginをインストールしたら、全部のSkillが使えるようになるの？

**短答**：**2つのPlugin（document-skills / example-skills）を選んでインストールする**。

```
/plugin marketplace add anthropics/skills
/plugin install document-skills@anthropic-agent-skills
/plugin install example-skills@anthropic-agent-skills
```

全Skillが一括で入るわけではなく、Plugin単位で選択する。document-skillsはClaude.aiの裏側でも実際に使われている本番品質のSkill。必要なものだけインストールするのがベスト。

---

### Q2-4. データ分析で金融データを扱う場合、セキュリティは大丈夫？

**短答**：**第2回「分ける」と第3回「防ぐ」で学んだ対策がそのまま適用される**。

- **DevContainer内で実行**: データがローカルPCに漏れない（第2回）
- **Hooksでネットワーク制限**: 外部へのデータ送信をブロック（第3回）
- **PIIマスキング**: Langfuseのトレースにも機密データが残らない（第6回）

API経由でモデルに送られるデータの扱いについてはAnthropicの利用規約を確認すること。

---

### Q2-5. SubAgentsってどういう時に使うべき？単一セッションじゃダメ？

**短答**：**「調査」「テスト」「ドキュメント」のように独立した作業を並列実行したい時**。

単一セッションの問題は2つ：
1. **コンテキスト汚染**: 調査結果がメインセッションのコンテキストを埋めてしまう
2. **直列実行**: 調査→テスト→ドキュメントを順番にやると時間がかかる

SubAgentsなら各タスクが独立コンテキストで動き、メインセッションはクリーンなまま。コスト面でも効率的（コンテキスト分離でトークン節約）。

---

### Q2-6. Dynamic Context型のSkillで !`command` 構文を使う場合、セキュリティリスクはある？

**短答**：**ある。信頼できないSkillではシェルコマンドが実行される可能性がある**。

対策：
- **信頼できるソースのSkillのみインストール**（anthropics/skills公式等）
- `"disableSkillShellExecution": true` を設定に追加すると、全Skillのシェル実行を無効化できる
- 第3回のHooksのDenyルールと併用する

---

### Q2-7. Routinesのスケジュール実行、具体的にどんな使い方が実用的？

**短答**：**3つの定番パターンがある**。

1. **週次ドキュメント更新**: マージ済みMRをスキャン → 古いドキュメントを検知 → 更新MRを自動作成
2. **デプロイ後検証**: CDパイプライン完了後にAPIエンドポイントで呼び出し → スモークテスト → 結果をSlack投稿
3. **夜間セキュリティスキャン**: 依存パッケージの脆弱性チェック → 新しいCVEが見つかったらIssue起票

いずれも「人間がやると忘れがちだが、やらないとリスクがある」定型作業。

---

### Q2-8. コードベース探索の「Code Intelligenceプラグイン」って入れるべき？

**短答**：**入れるべき**。「Go to Definition」「Find References」の機能がClaude Codeに追加される。

プラグインなしでもファイル検索やgrep検索は可能だが、大規模プロジェクトでは精度と速度が段違い。TypeScript/Python/Java/Go/Rust等に対応。`/plugins` からインストールできる。

**短答**：S9の全体フロー図がそのまま答えです。

### Q3-2. Agent TeamsとSubAgentsの違いをもう少し詳しく教えてください

**短答**：**第8回で詳しく扱うが、使い分けの要点を**。

| | SubAgents | Agent Teams |
|---|---|---|
| 構造 | Main → Sub（一方向） | P2Pメッセージング |
| 通信 | 結果報告のみ | 相互通信可能 |
| コスト | 低 | 高（3-4倍） |
| 適用場面 | 調査・テスト・分析 | FE/BE/Test連携の大規模リファクタ |

**判断基準**: エージェント間で「会話」する必要があるか。なければSubAgentsで十分。

---

### Q3-3. Skillをチームで共有するベストプラクティスは？

**短答**：**3段階のスコープを使い分ける**。

1. **個人**: `~/.claude/skills/` — 自分だけのワークフロー
2. **プロジェクト**: `.claude/skills/` にコミット — チーム全員がgit pullで使える
3. **Plugin化**: 複数プロジェクトで共有 — `/plugin` として配布

チームで最初にやるべきは「**プロジェクトレベルでSkillをコミットする**」。新メンバーがgit cloneした瞬間からチームのワークフローが使える。

---

### Q3-4. GitLabのIssueをClaude Codeから直接操作するにはどうする？

**短答**：**glab CLI（GitLab公式CLI）を使う**。

Claude Codeはターミナルで動くので、glab CLIが使える環境なら直接Issue操作が可能。CLAUDE.mdに「GitLabのIssue操作にはglab CLIを使用する」と書いておくとスムーズ。

---

### Q3-5. Skillのdescriptionを最適化するコツは？

**短答**：**3つのポイント**。

1. **「何をするか」と「いつ使うか」の両方を書く**
2. **少し「pushy」にする**: Claude Codeはスキルを「使わない」方向に偏りがちなので、発動条件を明示的に
3. **1,536文字以内に収める**: 重要な情報を前に持ってくる

Skill Creatorの⑥ステップ（description最適化）を使えば自動で最適化してくれる。
