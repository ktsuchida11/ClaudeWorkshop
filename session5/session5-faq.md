> 会場で確実に拾いたい問い: **Q1（SDD と TDD）・Q7（キャッシュ）・Q8（Opus Plan）・Q11（金融の managed 設定）**。

## ワークフローと5つの規律

### Q1. SDD（仕様駆動）と TDD（テスト駆動）は何が違う？両方やるの？

レイヤーが違うので両方やります。**SDD は「仕様」を単一の真実にする**やり方で、仕様からコード・テスト・ドキュメントを生み出します（アーキテクチャや非機能要件まで含む、広い枠）。**TDD は「失敗するテスト」を起点にコードを書く**、より狭い実装ループです。今日のワークフローでは、①で SDD（spec.md）、④で TDD（テスト）と、別フェーズに位置づけています。多くの SDD ワークフローは TDD のテストを成果物として生みます。

### Q2. TDD は Claude が勝手にやってくれる？

いいえ。Claude は放置すると**実装を先に**書きます。TDD にしたい時は「**失敗するテストを先に書いて。実装はまだしないで**」と明示してください。そのうえで Red→Green→Refactor を回します。Plan Mode で先にテスト計画だけ立てさせるのも有効です。

### Q3. Plan Mode は軽いタスクでも毎回使うべき？

いいえ。単純な機械的修正には不要です。**複数ファイル・リファクタ系・曖昧・本番に触る**作業で使ってください。Plan Mode 単体では足りないことが多いので、**Ask User** で不明点ゼロまで詰めてから spec.md にするのがコツです。

## 設計を残す（DDD）

### Q4. CLAUDE.md と Skills はどう使い分ける？両方いる？

役割が違うので両方使います。**CLAUDE.md** は常時ロード＝「どんな文脈でも守る共通ルール」だけ（200行以内、理想は100行）。**Skills** は必要な時だけ動的ロード＝個別の手順や型。迷ったら「毎回必ず要るか？」で判断し、そうでなければ Skill へ寄せます。CLAUDE.md の遵守率は約70%とされ、盛りすぎると中間の指示が無視されやすくなります。

### Q5. GitHub / GitLab は MCP と Skill のどちらで登録すべき？

どちらでも「**登録しておく**」ことが大事です。コミット・PR・決定（なぜこの実装か）が GitHub/GitLab に残ること自体が「② 設計を残す（DDD）」の一部になります。手軽さなら MCP コネクタ、コンテキスト効率と監査性なら Skill＋CLI（`gh` / `glab`）です。**金融など統制重視なら Skill＋CLI か、監査済みの MCP** を選びます。1タスク＝1コミット、コミットレビューといったワークフローがそのまま回るようになります。

## 計画して実行（サブエージェント）

### Q6. 組み込みサブエージェントは自分で起動するの？何をサブエージェントにすればいい？

多くは裏で Claude Code が自動起動します（特に explore は複数ファイル確認で多用）。**何を任せるかの基準は「独立して完結する単位」**です。具体的には、①**探索**（Explore＝読み取り専用でコードベース調査）②**実装**（plan.md の1タスク＝1エージェント＝1コミット）③テスト生成・局所リファクタ、など。探索が読み取り専用なので、メインのコンテキストが汚れにくいのが利点です。カスタム作成は、組み込み（explore / plan / general-purpose）で足りない時だけで十分です。

## コンテキスト・キャッシュ・モデル選択

### Q7. プロンプトキャッシュって結局なに？コンテキストとどう関係する？

毎回コンテキストは送り直しますが、モデルが一度読み込んだ「**処理済みの状態**」を一時保存して使い回すのがプロンプトキャッシュです。保存されるのは“文章”ではなく、送るコンテキストの**先頭**（System＋ツール定義＋CLAUDE.md＋これまでの会話）を処理した結果。**同じ先頭を、有効期間（既定 約5分）内に再送するとヒット**し、その分の入力コストは基本料金の約1割・高速になります。つまり「**コンテキストの作り方が、キャッシュの効き＝コスト/速度を決める**」という関係です。上の層（CLAUDE.md・ツール・モデル）を変えると先頭が変わり、そこから下は作り直し（満額）になります。

### Q8. Opus Plan（`model: opus-plan`）って何？いつ使う？

「**計画・仕様は Opus、実装の反復は Sonnet**」と役割でモデルを分ける設定です。深く考えたい設計・計画フェーズだけ賢いモデルを使い、繰り返しの実装は速くて安いモデルにする、という使い分けでコストを抑えます。1セッション内で頻繁にモデルを切り替えるとキャッシュが全再計算で重いので、**役割で分ける**のが効きます。

### Q9. CLAUDE.md を200行に収めるのがきつい。`/clear` はどのくらいで？

CLAUDE.md は「常にどんな文脈でも必要なルール」以外を Skills やサブエージェントに逃がし、**条件付きロード**（特定の拡張子の読み書き時だけ実行する指示）も使うと削れます。`/clear` の目安はコンテキストの **20〜30%**。膨らんだら一度クリアし、続けたい時は引き継ぎ用の Markdown（ハンドオーバー）を出力して新セッションで読ませます。脱線時は Escape→/clear。

## 金融機関での運用

### Q10. MCP と Skill＋API はどちらを使えばいい？

手軽さなら MCP、コンテキスト効率と統制なら Skill＋API です。MCP はツール定義が常にコンテキストを占有しがちですが、Skill＋API は必要な時だけ動的にロードされます。**金融機関では特に Skill＋API を優先**してください（外部サーバーを増やさず、自前コードで監査できる）。

### Q11. 金融機関では外部 MCP/プラグインは一切使えない？managed-settings は個人でも？

「一切禁止」ではなく「**管理された範囲で**」が答えです。**managed-settings.json / managed-mcp.json** で許可リスト運用にし、導入前にソースレビュー・バージョンピン・最小権限を徹底すれば使えます。組織は MDM／Intune／GPO で配布してユーザーが上書きできないようにしますが、自分の端末に手動で置くこともできます。**deny（拒否）は常に allow（許可）に優先**し、`allowManagedMcpServersOnly: true` で許可リストのサーバーだけ起動できます。説明文に隠し指示を仕込む **tool poisoning** に注意してください。

## この先

### Q12. 複数エージェントはいつから？次いつやる？

今日の「型」（5つの規律＋コンテキスト設計）が回るようになってからで十分です。1つのコンテキストを管理できない状態で並列化すると、**間違ったものを高速で量産する**だけになります。これは成熟度の**フェーズ3**（複数エージェント／ループ）にあたり、第6回（ログ/トレース）の後に、ハーネス・Agent Teams・ループを実ユースケースと一緒に紹介する予定です。
