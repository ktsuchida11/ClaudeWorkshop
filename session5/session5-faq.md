# 第5回「AIとの仕事の型」 FAQ予想問答集

質疑応答5分間に備えた想定問答集。第5回のテーマは「**【残す】への橋渡し ― 5つの防止アクションと3段階成熟度モデル**」。

3層構造で整理：

- **L1 未経験者から出そうな質問**：5つの防止アクション・SDD・成熟度の基本概念
- **L2 日常利用者から出そうな質問**：実践的な運用・トレードオフ
- **L3 深掘り質問**：組織展開・SDD/AIDLCの比較・限界

---

## L1：未経験者から出そうな質問

### Q1-1. 5つの防止アクション、全部やらないとダメですか？最低限はどれ？

**短答**：**最低限は①要件を明確にする + ⑤レビューする**の2つ。この2つだけで「何を作るか分からないまま始める」「AIの出力を無検証でマージする」という最大の2つの罠を防げる。

残り3つは段階的に足していけばいい。②ドキュメント → ③計画 → ④テストの順で追加するのが自然。全部を一気に完璧にやろうとすると逆に手が止まる。**今日から①と⑤だけ始める**のが現実的。

---

### Q1-2. SDD って、要するに「仕様書を先に書け」ということですか？

**短答**：**その通り**です。ただし従来の仕様書作成と決定的に違うのは、仕様書を書く作業自体もAIに手伝ってもらえること。「こういう機能を作りたい」と伝えれば、Claude Codeが仕様書のドラフトを作ってくれる。人間はそれをレビューして修正する。

SDDの本質は「仕様書を書くのが大変」ではなく「**仕様書をSingle Source of Truthにして、そこからコードを生成させる**」という発想の転換。仕様が曖昧なまま「とりあえず作って」と頼むのをやめるだけで、手戻りが激減する。

---

### Q1-3. バイブコーディングでも十分うまくいってるんですが、SDDに移行すべきですか？

**短答**：**場面による**。個人ツールやプロトタイプなら、バイブコーディングで十分。「動けばいい」「自分だけが使う」なら無理にSDDに移行する必要はない。

SDDが必要になるのは以下の場面：
- **チームで開発する**（他の人が理解できる必要がある）
- **本番環境で運用する**（品質・保守性が重要）
- **3ヶ月後の自分が読む**（設計意図が残っていないと詰む）

今日のスライドで「バイブコーディングが悪いのではない」と言ったのはこの理由。**自分の現在地と目的に合わせて選ぶ**のが正しい。

---

### Q1-4. 「AIとの合意形成」って具体的にどうやるんですか？

**短答**：**3ステップ**で進めます。

1. **要件を書き出す**：「こういう機能を作りたい」をdocs/spec.mdに自然言語で書く
2. **Claude Codeに確認させる**：「このspec.mdを読んで、不明点や曖昧な点があれば質問して」と頼む
3. **質問に答えて仕様を固める**：Claude Codeからの質問に答えることで、暗黙の前提が明文化される

人間同士なら「空気を読む」ことで補えるが、AIは空気を読まない。だから**書いて渡す**のが基本。逆に言えば、書けば書くほど正確に動く。

---

### Q1-5. レビューが最重要と言いましたが、AI生成コードのレビューって何を見ればいいですか？

**短答**：**3つの優先順位**で見ます。

1. **データ永続化**（最優先）：DB操作、ファイル書き込み、API呼び出しの結果保存。ここが壊れると本番障害になる
2. **ビジネスロジック**：計算式、条件分岐、バリデーション。普通のコードレビューと同じ目で読む
3. **UI/表示周り**（許容度高）：些細なバグは許容範囲。後から直せるものは後で直す

**全行を読む必要はない**。1回の作業量を小さくしてレビュー範囲を絞り、クリティカルな部分に集中する。AIに「このコードの変更点を要約して」と頼んで下読みさせるのも有効。

## L2：日常利用者から出そうな質問

### Q2-1. SDD のステアリングファイルと CLAUDE.md の違いがよく分からない

**短答**：**更新頻度と寿命が違います**。

| | CLAUDE.md | ステアリングファイル |
|---|---|---|
| 寿命 | プロジェクト全体 | 作業バッチごと |
| 内容 | ルール・規約・構造 | 作業指示・タスク・進捗 |
| 更新頻度 | たまに | 毎日 |
| 例 | 「ESLintを使う」「Conventional Commits」 | 「今回はユーザー認証機能を実装する」 |

CLAUDE.mdは「このプロジェクトのルールブック」。ステアリングファイル（docs/plan.md等）は「今回の作業の指示書」。CLAUDE.mdに「docs/plan.mdを参照して作業を進めて」と書いておくのがベスト。

---

### Q2-2. AIDLCのInception→Construction→Operationって、具体的にどう運用するんですか？

**短答**：3フェーズの実務的な運用は以下の通り。

**Inception（構想）**：
- 要件を定義し、設計方針を決める
- PRD（プロダクト要件定義書）を作成
- **ゲート**：人間が要件をレビューし「この方向性でOK」と承認

**Construction（構築）**：
- 承認された要件に基づいてClaude Codeが実装
- テストコードも一緒に生成
- **ゲート**：人間がコードレビュー + テスト結果を確認

**Operation（運用）**：
- デプロイ後の監視・評価
- 問題があればInceptionに戻る
- **第6回「ログ/トレース」**でここの実装方法を扱う

**ポイント**：各ゲートで「人間が承認しないと次に進めない」という制約が、品質を担保する仕組み。

---

### Q2-3. 第4回の構造化ノート（docs/plan.md等）と今日のSDD、どう繋がるんですか？

**短答**：**第4回の構造化ノートはSDDの「ステアリングファイル」そのもの**です。

第4回では「コンテキストウィンドウの中身は揮発する。ファイルは残る」という原則で、作業内容をファイルに書き出すことを学んだ。今日はその**なぜ**と**体系**を学んだ。

| 第4回で学んだこと | 第5回で体系化 |
|---|---|
| docs/plan.md | → SDD のステアリングファイル |
| docs/decisions.md | → 設計判断の記録（SDD永続ドキュメント） |
| docs/progress.md | → 作業ログ（運用フェーズの記録） |
| 「ファイルに残す」 | → 「ドキュメントを残す」（防止アクション②） |

第4回は「HOW（どうやって残すか）」、第5回は「WHY（なぜ残すのか）+ WHAT（何を残すのか）」。

---

### Q2-4. タスク分解の粒度はどのくらいが適切？

**短答**：**「1回のClaude Code実行で完了し、差分がレビュー可能な大きさ」**が目安。具体的には：

- **ファイル変更数**：3-5ファイル以内
- **コード行数**：追加/変更が100-300行以内
- **時間の目安**：Claude Codeの実行が5-15分で完了する規模

大きすぎると：レビューが大変 / コンテキストウィンドウが圧迫される / 失敗時のロールバックが困難
小さすぎると：オーバーヘッドが増える / 全体像が見えにくい

**迷ったら小さく分ける方を選ぶ**。大きいタスクは後から分割できないが、小さいタスクは後から結合できる。

---

### Q2-5. Claude Codeにテストコードを一緒に生成させるとき、どう指示すればいい？

**短答**：CLAUDE.mdに以下を書いておくのが最も確実。

```
# テストルール
- 新しい機能を実装する時は、必ず対応するテストも一緒に書くこと
- テストは jest / pytest 等のフレームワークを使うこと
- エッジケース（null、空文字、境界値）のテストを必ず含めること
- テストが全てパスすることを確認してから完了とすること
```

ポイントは**「テストも書いて」ではなく「テストも一緒に書くこと」とルール化**すること。CLAUDE.mdに書けば毎回の指示が不要になる。さらに第3回で学んだHooksを使えば、テストが失敗した場合にコミットをブロックすることも可能。

---

### Q2-6. 実演で見せた4ステップ、実務では毎回やるんですか？時間かかりすぎませんか？

**短答**：**毎回やりますが、慣れると5分で終わる**。

最初は面倒に感じるが、以下の理由で結果的に時間を節約できる：
- **手戻りが減る**：仕様を先に決めるので「思ったのと違う」が起きにくい
- **レビューが速い**：計画があるので何をチェックすべきか明確
- **記録が残る**：3ヶ月後に「なぜこうしたか」を調べる時間がゼロになる

実務ではspec.mdとplan.mdは**前回の作業のコピー＆修正**で作ることが多い。テンプレート化すれば「埋めるだけ」になる。

---

### Q2-7. 場当たり的な実装で技術的負債が蓄積した既存プロジェクト、今からSDD化できる？

**短答**：**完全なSDD化は不要で、今後の変更から適用するだけでいい**。

具体的には：
1. **今すぐ**：CLAUDE.mdを作って基本ルールを書く（30分）
2. **次の変更から**：変更内容をdocs/plan.mdに書いてから作業する（5分追加）
3. **重要な判断時**：docs/decisions.mdに記録する（2分追加）
4. **段階的に**：既存コードのドキュメントは触る時に書く（Boy Scout Rule）

全部を一気にドキュメント化しようとすると破綻する。「今後触る部分だけ、触る時に整理する」のが現実的。

## L3：深掘り質問

### Q3-1. SDD と AIDLC、うちのチームにはどっちが合いますか？

**短答**：**チーム規模と組織のガバナンス要件で判断**します。

| | SDD | AIDLC |
|---|---|---|
| チーム規模 | 1-5人 | 5人以上 |
| ガバナンス | 緩め（チーム内の信頼で運用） | 厳格（承認フローが必要） |
| ドキュメント量 | 中（PRD + 設計書 + ステアリングファイル） | 多（フェーズゲートの記録が追加） |
| 適した組織 | スタートアップ、小規模チーム | エンタープライズ、規制業界 |
| 始めやすさ | 今日から始められる | 組織の承認フロー整備が先 |

**ほとんどのチームはSDDから始めるのが正解**。AIDLCはSDDを運用した上で「もっとガバナンスが必要」と感じた時に検討する。

---

### Q3-2. 「AI出力を無条件に信頼しない」と言うけど、AIの出力の何%くらいが問題なんですか？

**短答**：**統計は出しにくいが、体感で10-20%に何らかの問題がある**。問題の内訳：

- **動くが非効率**（5-10%）：O(n²)のアルゴリズムを使う、不要なDB呼び出し等
- **エッジケース漏れ**（5-10%）：null処理忘れ、境界値バグ等
- **セキュリティ問題**（1-3%）：SQLインジェクション、XSS、認証バイパス等
- **設計方針との不整合**（3-5%）：動くが既存のアーキテクチャと合わない

重要なのは**「動いた≠正しい」**という認識。構文エラーはほぼゼロだが、品質面の問題は人間がレビューしないと見つからない。だから⑤レビューが最重要。

---

### Q3-3. 今日の5つの防止アクション、第1回〜第4回で学んだ機能とどう対応する？

**短答**：これまでの回で学んだ機能が、5つの防止アクションを**技術的に支える**関係にあります。

| 防止アクション | 支える機能（学んだ回） |
|---|---|
| ① 要件を明確にする | CLAUDE.md（第3回）、構造化ノート（第4回） |
| ② ドキュメントを残す | 構造化ノート（第4回）、コンテキストエンジニアリング戦略2（第4回） |
| ③ 計画を立てる | Skills（第4回）、スラッシュコマンド（第4回） |
| ④ テストする | Hooks PreToolUse（第3回）、サブエージェント（第4回） |
| ⑤ レビューする | Permission（第3回）、Sandbox（第3回）、サブエージェント（第4回） |

**第1〜4回は「壁とツール」、第5回は「仕事の進め方」**。両方揃って初めて安全で効率的なAI開発になる。

---

### Q3-4. AIDLC の「ゲート」って、CI/CD に組み込めますか？

**短答**：**GitHub Actionsと組み合わせれば実装可能**です。

- **Inceptionゲート**：PRDが`docs/prd.md`に存在しないとPRを作成できないルール
- **Constructionゲート**：テストがパス + コードレビュー承認が必要（GitHub Branch Protection）
- **Operationゲート**：デプロイ前にステージング環境でのスモークテスト必須

第3回で学んだHooksと組み合わせれば、Claude Code側でも「plan.mdが存在しないと実装を開始しない」というガードを設定できる。**第7回「ユースケース」でGitHub Actions連携を扱う予定**。

---

### Q3-5. 今日の成熟度モデル、Level 2（SDD）にいる場合、何を目安にLevel 3に進むべき？

**短答**：**3つのシグナルが出たら検討する**。

1. **チームが5人を超えた**：暗黙の合意では品質が維持できなくなる
2. **規制対応が必要になった**：金融・医療・公共系で監査証跡が求められる
3. **リリース頻度が上がった**：週次以上のリリースで、レビューの抜け漏れが発生し始めた

逆に言えば、これらのシグナルがなければ**SDDで十分**。AIDLCはオーバーヘッドが大きいので、必要になるまで導入しない方が良い。

---

### Q3-6. 今日のレビューの話、GitHub Copilot Code Review等のAIレビューツールと併用できる？

**短答**：**併用可能で、むしろ推奨**。AIレビューと人間レビューは得意分野が違う。

| | AIレビュー | 人間レビュー |
|---|---|---|
| 得意 | パターンマッチ（未使用変数、型エラー） | ビジネスロジックの妥当性 |
| 得意 | セキュリティパターン（SQLインジェクション等） | 設計方針との整合性 |
| 苦手 | 「この実装は要件と合っているか」 | 大量コードの網羅的チェック |

運用パターン：**AIレビューで機械的な問題を先に潰す → 人間レビューでビジネスロジックに集中**。今日のスライドで「AIに下読みさせることも有効」と言ったのはこの使い方。

---

### Q3-7. 「速く作れる」と「正しく作れる」の両立は本当にできるんですか？

**短答**：**SDDを使えば両立できる、というのが第5回の結論**です。

バイブコーディングは「速いが正しさが保証されない」。従来のウォーターフォールは「正しいが遅い」。SDDはその中間で「仕様を先に決めるからAIが正しく速く作れる」。

具体的な数字感：
- バイブコーディング：実装10分 + 手戻り30分 = 40分
- SDD：仕様5分 + 実装10分 + レビュー5分 = 20分

仕様を書く5分が、手戻り30分を節約する。**型は速度を殺すのではなく、速度を安定させる**。
