# Claude Code 勉強会 第5回「AIとの仕事の型」スライド仕様書

## 全体方針

- **テーマ**: 確立した開発規律（**SDD＝仕様駆動開発 / DDD＝ドメイン駆動設計 / TDD＝テスト駆動開発 / 計画→実行**）を、Claude Code の機能で**ワークフロー**にする。個々の機能を「束ねて」タスクを完遂する、複数エージェント開発の前段階。
- **背骨**: 「機能を束ねる5つの規律」＝ ①要件(SDD) → ②設計を残す(DDD) → ③計画→実行 → ④テスト(TDD) → ⑤レビュー。各規律を Claude Code のどの機能で実現するかを示す。
- **横断の軸**: コンテキスト最適化・モデル選択（Opus Plan）・プロンプトキャッシュ。ワークフロー全体のコストと速度の土台。
- **流れ**: 振り返り → 現在地（成熟度の3フェーズ）→ やりがちな罠 → 横断の軸（最適化・モデル選択・キャッシュ・4層）→ 5規律ワークフロー（全体像→設計→計画実行→品質）→ 金融での調達 → 実演 → まとめ → 次回予告。
- **枚数**: 本編15枚 ＋ 付録2枚（KEY TAKEAWAY は番号なしの14枚目）。
- **デザイン**: Midnight Executive。ネイビー基調＋アイスブルー＋レッドのアクセント。アクセントの帯・下線は使わず、カード＋アイコン円＋テイクアウェイ吹き出しで構成。

### S5: 今日の軸・横断 ― コンテキスト最適化とモデル選択（5–7分）★重要
- タイトル: コンテキスト最適化とモデル選択 ― ワークフロー全体に効く
- 左（コンテキスト最適化）: 有限で右肩上がり。開始時に CLAUDE.md/MCP が 5〜10% 占有すると後が効かない。入出力にコスト＝最適化はコスト最適化。
- 右（モデル選択＝Opus Plan）: 計画・仕様は Opus、実装の反復は Sonnet、簡単な探索は Haiku。1セッション内の切替は重い→役割で分ける。`model: opus-plan`（計画=Opus / 実装=Sonnet）。
- テイクアウェイ: コンテキストは設計対象、モデルは作業で使い分け。これがワークフロー全体の土台。

### S6: プロンプトキャッシュとは ― 何を保存し、いつ効くのか（7–9分）
- タイトル: 何を保存し、いつ効くのか ― 新しい概念を具体例で
- 冒頭: 毎回コンテキストは送り直す。が、モデルが一度読んだ「処理済みの状態」を一時保存して使い回す仕組み。
- 左（何を保存している？）: 送るコンテキストの**先頭**（System＋ツール定義＋CLAUDE.md＋会話）を、モデルが読んだ**処理済みの状態**として保存。"文章"ではなく"処理結果"。有効期間は既定 約5分（使うたび延長）。
- 右（いつ効く？・具体例）: 毎回 CLAUDE.md＋会話を送る＝先頭が同じ。1回目＝全部前処理（満額・遅い）、2回目以降（5分以内）＝先頭は読むだけ→**コスト約1/10・高速**。効く条件＝同じセッションで連続、先頭を変えない。
- テイクアウェイ: ヒット＝コスト約1割＋高速。ミス（先頭を変える／5分超／モデル切替）＝先頭から全部作り直し（満額）。
- ねらい: 新概念なので「何を保存し・いつ効くか」を具体例で丁寧に説明する。

### S7: コンテキストの4層モデル ― どこを変えると作り直しになるか（9–10分）
- タイトル: どこを変えると作り直しになるか（再計算コスト）
- 冒頭: 前ページの通り、キャッシュは「先頭から」効く。だから上の層を変えると、そこから下が全部作り直し＝コスト増・遅延。
- 4層（上＝先頭/最古、下＝最新）: ①**System**（モデル。切替で全再計算）②**Tools**（MCP・プラグイン。追加/削除・更新でここから下が再計算）③**CLAUDE.md / Skills**（変更・スキル追加でここから下が再計算。常時ロードはこの層）④**Message**（会話。末尾だけなので最も軽い）。
- 右の注釈: 再計算コスト 大（上を変えるほど広い）/ 小（末尾だけ）。
- テイクアウェイ: 失われるのは記憶ではなくキャッシュ（既定TTL約5分）。モデル切替は全再計算→モデルは役割で分ける（Opus Plan）。

### S8: タスク完遂のワークフロー（全体像）（10–12分）★重要
- タイトル: 確立した規律を、Claude Code の機能で束ねる
- 5規律＝ワークフロー（各：規律／手法／主な機能）:
  1. **要件** ／ 仕様駆動(SDD) ／ Plan Mode＋Ask User → spec.md
  2. **設計を残す** ／ ドメイン駆動(DDD) ／ CLAUDE.md・Skills に文書化
  3. **計画→実行** ／ 計画して実行 ／ plan.md → サブエージェント実装
  4. **テスト** ／ テスト駆動(TDD) ／ 失敗するテストを先に書く
  5. **レビュー** ／ 品質ゲート ／ /code-review ＋ 人間
- テイクアウェイ: 仕様→計画→実装→テスト→レビュー。仕様が単一の真実、各ステップ間に人間。次の3枚で各フェーズを掘る。

### S9: ワークフロー詳細①｜設計 ― 要件(SDD) と 設計の文書化(DDD)（12–14分）★重要
- タイトル: 要件(SDD) と 設計の文書化(DDD) ― 書いてから作る
- 左（① 要件 ＝ SDD・仕様駆動開発）: Shift+Tab で Plan Mode／explore（読み取り専用）でコードベース把握／Ask User で不明点ゼロまで／**docs/spec.md** に保存＝仕様が単一の真実。
- 右（② 設計を残す ＝ DDD・ドメイン駆動設計）: チームと AI で目的・設計を共有・文書化／**何を**：用語・設計判断・なぜそうしたか／**どこに**：CLAUDE.md（200行）＋詳細は Skills/docs／**GitHub / GitLab を MCP か Skill に登録**：コミット・PR・決定を残す。
- テイクアウェイ: ズレたら「仕様」を直す（チャット履歴ではなく）。仕様とドメインモデルが、以降の全工程の拠り所になる。

### S10: ワークフロー詳細②｜計画してから実行 ― plan → execute（14–16分）★重要
- タイトル: plan → execute ―「作る工程」をサブエージェントで自動化
- 左（計画・実行の前に）: 仕様を**番号付きタスク**に分解／**docs/plan.md** に範囲・完了基準／計画は Opus で深く（Opus Plan）。
- 右（実行・何をサブエージェントに）: **独立して完結する単位**を渡す。**探索**：Explore（読み取り専用）でコードベース調査。**実装**：plan.md の1タスク＝1エージェント＝1コミット。テスト生成・局所リファクタなども委譲できる。
- テイクアウェイ: 計画は実行の前、実装はサブエージェントに委譲。コンテキストが崩れたら `/clear` して仕様から再開する。

### S11: ワークフロー詳細③｜品質 ― テスト(TDD) と レビュー（16–18分）★重要
- タイトル: テスト(TDD) と レビュー ― 動く ≠ 正しい
- 左（④ テスト ＝ TDD・テスト駆動開発）: Claude は放置すると実装を先に書く→**「失敗するテストを先に」と明示**／Red→Green→Refactor／エッジ・性能・脆弱性の穴。
- 右（⑤ レビュー・品質ゲート）: 各ステップ間に人間ゲート／`/code-review` で下読み→最後に通し／**データ永続化**が最クリティカル／仕様がズレたら仕様を直す。
- テイクアウェイ: テストは AI の「明確な的」。5フェーズが回れば、作業を他人にも複数エージェントにも渡せる。

### S12: 金融機関で使うなら（18–19分）★重要
- タイトル: 2つだけ具体的に：MCP は慎重に・設定は管理者が固める
- 左（① MCP を使うのは慎重に・サプライチェーン）: 公開 MCP は玉石混交・**tool poisoning**（説明文の隠し指示）／承認済みサーバーへの不正アップデートも侵入経路／導入前に監査＋バージョンピン＋最小権限／外部 MCP より自前の Skill＋API を優先。
- 右（② 設定は管理者側で固める）: **managed-settings.json / managed-mcp.json** で集中管理／**MDM・Intune・GPO** で配布（ユーザー上書き不可）／許可リストのみ起動・deny 優先／マーケットプレイスは strictKnownMarketplaces で限定。`allowManagedMcpServersOnly: true`。
- テイクアウェイ: 鉄則は「監査して・固定して・最小権限で」。便利さは管理された範囲の中で取りに行く。

## 付録（実践TIPS集・2ページ）

### Appendix 1/2: コンテキスト最適化と、タスク完遂のワークフロー
- **A. コンテキスト最適化・モデル選択（S5–S7）**: /context で定期点検（未使用 MCP は無効化）／20〜30% で /clear（脱線時は Escape→/clear）／ハンドオーバー運用（引き継ぎ Markdown）／CLAUDE.md は200行以内／Opus Plan（model: opus-plan、計画=Opus/実装=Sonnet）／モデル切替は役割で（同一セッション内は全再計算）。
- **B. 5つの規律＝ワークフロー（S8–S11）**: ① 要件(SDD)＝Plan Mode＋explore＋Ask User → spec.md／② 設計を残す(DDD)＝CLAUDE.md・Skills＋GitHub/GitLab(MCP/Skill)に残す／③ 計画→実行＝plan.md → サブエージェントで1タスク=1コミット／④ テスト(TDD)＝失敗するテストを先に。Red→Green→Refactor／⑤ レビュー＝/code-review＋人間ゲート、永続化を重点／ズレたら仕様を直す（チャット履歴ではなく spec を修正）。

### Appendix 2/2: 金融での運用と、コスト
- **C. 金融機関で安全に運用する（S12）**: MCP は慎重に（玉石混交・tool poisoning・不正アップデート）／導入前に監査＋バージョンピン＋最小権限／外部 MCP より Skill＋API を優先／managed-settings・managed-mcp を MDM・Intune・GPO で配布／allowManagedMcpServersOnly（許可リストのみ・deny 優先）／strictKnownMarketplaces。
- **D. コストを意識する（横断）**: 入力・出力にコスト。Opus は Sonnet の約5倍／effort 使い分け（複雑=Opus／実装=Sonnet／簡単=Haiku）／4層キャッシュを壊さない（モデル切替・MCP入替は重い）／既定TTL約5分（間が空くと再計算）／/clear と引き継ぎで長セッションのコストを抑える。

## 用語
- **SDD ＝ 仕様駆動開発（Spec-Driven Development）**: 仕様が成果物、コードはビルド出力。仕様からテスト・コード・ドキュメントを生成する。
- **DDD ＝ ドメイン駆動設計（Domain-Driven Design）**: ドメインの言葉・設計を共有・文書化する。CLAUDE.md / Skills / GitHub に残す。
- **TDD ＝ テスト駆動開発（Test-Driven Development）**: 失敗するテストを先に書き、通す→整える（Red→Green→Refactor）。

## キーファクト（話す際の根拠・2026年6月時点）
- SDD は 2025年（GitHub Spec Kit 9月）に確立し、2026年には Claude Code を含む主要AIコーディングツールが各々の実装を持つ。仕様が単一の真実。
- Claude Code の標準SDDワークフロー（コミュニティで定番化）＝ 機能説明→Plan Mode→仕様起こし→Ask User→spec.md保存→サブエージェントで1タスク=1コミット→コミットレビュー→仕様修正→/clear再開→最終レビュー。
- TDD は Claude が放置すると実装を先に書くため「失敗するテストを先に。実装はまだ」と明示が必要。
- プロンプトキャッシュ既定TTLは約5分（2026年3月に1時間→5分へ戻った。クライアントが自動決定）。キャッシュ読み出しは基本料金の約1割。先頭（System/Tools/CLAUDE.md/会話）が一致し、TTL内で再送されるとヒット。tool_choiceや画像の有無の変化、モデル切替で無効化。
- 金融: managed-settings.json / managed-mcp.json を MDM/Intune/GPO で配布しユーザー上書き不可に。allowManagedMcpServersOnly:true で許可制（deny優先）。tool poisoning（説明文の隠し指示）に注意。
